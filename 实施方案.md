# 文档说明

本文档用于指导 STM32G474 无感 FOC 电机控制项目的实际开发、调试、测试和交付。它不是系统架构说明，而是类似建筑行业“施工组织设计、专项施工方案、质量验收方案、调试方案”的工程实施文件。

本方案适用于以下项目形态：

- STM32G474 主控；
- 低压 PMSM 电机；
- 三低边电流采样；
- TIM1 中心对齐三相互补 PWM；
- ADC2 注入组同步采样三相电流；
- 约 16kHz 高频 FOC 控制；
- ALIGN + OPENLOOP + Observer/PLL + Switchover + RUN 的无感启动链路；
- RUN 阶段采用 Observer 角度、电流环和速度环闭环运行。

本文档的核心目标是把项目从“能转起来”推进到：

```text
能解释；
能复现；
能保护；
能调试；
能交接；
能测试；
能扩展；
能长期运行。
```

# 01_项目实施总原则

## 01.01 阶段门禁原则

本项目采用阶段门禁机制。每个阶段必须具备明确的进入条件和退出条件。某一阶段没有通过，禁止继续向后叠加功能。

典型门禁如下：

```text
PWM 未验证，禁止接电机；
ADC 电流采样未验证，禁止闭电流环；
电流环未稳定，禁止调 Observer；
Observer 未稳定，禁止 Switchover；
Switchover 未稳定，禁止速度环强接管；
保护未验证，禁止长时间运行；
长时间运行未验证，禁止版本冻结。
```

这样做的目的，是避免问题叠加。电机控制项目最怕的是采样、电流环、Observer、速度环、补偿、保护一起改，最后所有问题混在一起，无法定位。

## 01.02 单变量调试原则

每次调试只允许修改一个主要变量。比如速度上不去时，不能同时修改 PI 参数、Observer 参数、前馈增益、死区补偿、采样点、电流限幅和 Switchover 条件。

每次修改必须记录：

```text
修改前参数；
修改后参数；
修改原因；
测试工况；
测试日志；
现象变化；
结论；
是否保留修改。
```

## 01.03 先验证测量，再验证控制

FOC 项目的调试顺序必须是：

```text
先确认 ADC raw 正确；
再确认电流换算正确；
再确认 Clarke/Park 正确；
再确认 PI 正确；
再确认 SVPWM 正确；
再确认 Observer 正确；
再确认 Switchover 正确；
再确认速度环正确；
最后再确认补偿和保护。
```

任何控制问题，优先怀疑测量链路。因为电流采样、母线电压、角度和速度这些基础量错误时，后面的控制算法即使公式正确，也会表现为错误。

## 01.04 安全优先原则

本项目涉及三相逆变桥、电机、电源、电流采样和 PWM 输出，安全优先级高于功能开发。

禁止以下行为：

```text
不验证 PWM 互补和死区就接电机；
不验证驱动使能默认状态就上电；
不验证过流保护就长时间运行；
不限流直接做堵转测试；
不看电流直接提高 iq_ref；
不看母线电压直接提高转速；
不验证采样点就开启高速闭环；
不记录 fault 原因就反复重启。
```

## 01.05 研发版本和产品版本分离

研发阶段允许：

```text
UART 打印较多；
参数写死在 foc_config.h；
部分保护先记录不触发；
使用按键控制启动停止；
使用 DebugSnapshot 辅助调试；
使用 DAC 输出关键变量。
```

产品阶段必须补齐：

```text
参数 Flash 存储；
参数合法性检查；
故障码体系；
故障快照持久化；
硬件 Break 链路；
上位机协议；
出厂标定；
版本管理；
回归测试。
```

# 03_阶段0：需求定义与边界冻结

## 阶段目标

明确本项目要做成什么、不做什么、做到什么程度算完成。

当前项目不能只定义为“让电机转起来”。工业级项目必须明确控制对象、电压范围、电流范围、速度范围、启动方式、采样方式、保护等级、调试方式和交付形态。

## 当前建议项目目标

```text
控制对象：低压 PMSM 电机；
主控芯片：STM32G474；
母线电压：标称 24V；
目标最高转速：3000rpm；
控制方式：PMSM FOC；
位置传感器：无位置传感器；
启动方式：OFFSET_CALIB -> ALIGN -> OPENLOOP -> Observer Switchover -> RUN；
电流采样：三低边电阻采样；
PWM：TIM1 中心对齐三相互补 PWM；
高频控制周期：约 16kHz，即 62.5us；
角度估算：Flux Observer + PLL；
运行阶段：Observer 角度闭环 + 电流环 + 速度环；
当前项目定位：工程研发验证平台，不直接定义为量产固件。
```

```text

极对数是否确认？
最大安全电流是多少？
电源是否有限流？
驱动器是否有硬件保护？
板子是否具备 FAULT 输入？
TIM1 BKIN 是否已接？
是否需要正反转？
是否需要弱磁？
是否需要堵转保护？
是否需要温度保护？
是否需要上位机参数调试？
是否需要 Flash 参数保存？
```

## 通过标准

```text
项目目标写入 README 或 PROJECT_SPEC.md；
关键硬件参数写入 PARAMETERS.md；
目标转速、目标电流、母线电压范围明确；
```

# 04_阶段1：工程基线整理与代码冻结

## 阶段目标

建立一个干净、可编译、可追踪的工程基线。在工业项目中，不能一边调硬件一边让代码结构不断漂移。必须先冻结一个可追踪版本。

## 开发内容

```text
整理 Keil 工程文件；
确认实际参与编译的 .c 文件；
删除或隔离重复目录；
确认 /Control 与 /Drivers/Control 是否存在重复；
确认 main.c、stm32g4xx_it.c、bsp_adc.c、bsp_tim.c 没有重复句柄定义；
确认 hadc1、hadc2、htim1、huart 等全局对象只有一处定义；
确认所有新增文件加入 Keil 工程；
确认编译无 Error；
确认 Warning 可解释。
```

## 推荐目录结构

```text
App/
  app_foc.c
  app_debug.c

Control/
  foc_sampling.c
  foc_pwm.c
  foc_current_loop.c
  foc_openloop.c
  foc_observer.c
  foc_switchover.c
  foc_deadtime.c
  foc_protection.c
  foc_types.h

Core/
  main.c
  stm32g4xx_it.c
  foc_config.h

Drivers/BSP/
  bsp_adc.c
  bsp_tim.c
  bsp_usart.c
  bsp_gpio.c

```

如果存在旧目录：


必须明确它是否参与编译。如果不参与编译，应改名为：

```text

```

或者直接删除，避免 Codex、Cursor 或其他同事误改旧文件。

## 必测项目

```text
全量编译；
Clean 后重新编译；
检查 map 文件；
检查重复符号；
```

# 05_阶段2：硬件资源确认与上电检查

```text

MCU 可烧录；
电源具有限流功能；
首次上电不接电机；
驱动母线电源可单独控制；

```

## 必查硬件资源

```text
MCU 型号：STM32G474；
U 相高边：TIM1_CH1；
V 相高边：TIM1_CH2；
W 相高边：TIM1_CH3；
U 相低边：TIM1_CH1N；
```

```text

不接电机时无异常电流；
```

## 不通过处理

```text
禁止进入 PWM 验证；
```

## 本阶段交付物
