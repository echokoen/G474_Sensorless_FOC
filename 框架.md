- **物理目录按模块细分**：便于源码边界、include path 和后续维护。

- **文件名表达模块职责**：例如 `foc_current_sense.c` 表示电流采样模块，`foc_svpwm.c` 表示 SVPWM 调制模块。

- **App/app_debug.c/.h**：低频调试输出入口，读取 RuntimeSnapshot 和 HfTimingSnapshot 后进行 UART 打印，原则上不直接接触高频控制细节。
- **Control/Common/foc_types.h**：公共类型中心，在 Keil 中归入 MotorControl 分组，定义 FOC_StateTypeDef、RuntimeSnapshot、TraceSnapshot、FaultSnapshot、ObserverSnapshot、CurrentSenseSnapshot、CurrentLoopSnapshot 等跨模块数据结构。
- **Control/Sensing/foc_current_sense.c/.h**：采样模块，负责 ADC 原始值读取、电流零偏校准、电流/母线电压换算、可信两相重构、fallback/hold-last 诊断。
- **Control/Modulation/foc_svpwm.c/.h**：SVPWM 与 PWM 输出模块，负责扇区计算、T1/T2/T0、三相 CCR、ADC 触发点、采样窗口诊断。
- **Control/Loop/foc_current_loop.c/.h**：电流环模块，负责 Clarke/Park、Id/Iq PI、Iq-only、DSoft、前馈/解耦、dq 电压限幅、InvPark。
- **Control/Startup/foc_startup.c/.h**：开环启动模块，负责对齐 tick、机械频率爬坡、电角度推进、开环速度输出。
- **Control/Observer/foc_observer.c/.h**：Observer 封装模块，负责调用 foc_flux_observer_pll，并输出估算角度、速度、角差、锁定状态和调试量。
- **Control/Startup/foc_handover.c/.h**：开环到 Observer 接管模块，负责 ready 判定、hold、blend、回退和 fail_reason。
- **Control/Modulation/foc_deadtime_comp.c/.h**：死区补偿模块，按三相电流方向生成 alpha/beta 电压修正，并处理过零平滑。
- **Control/Safety/foc_protection.c/.h**：基础软件保护模块，负责过流、母线欠压/过压、采样 hold 等检查，并提供统一停机入口。
- **Config/foc_config.h**：集中配置入口，包含硬件采样比例、电机参数、PI、启动、Observer、Handover、速度环、前馈、保护和调试开关。

# 01_控制系统总体架构

## 01_系统目标与边界

本系统目标是实现一套基于 STM32G474 的低压 PMSM 无位置传感器 FOC 控制平台。当前工程的重点不是做一个展示用 demo，而是建立一条可以继续产品化的主控制链路：从三相采样、坐标变换、电流环、SVPWM，到开环启动、Observer/PLL 估算、Startup/OpenLoop 到 Observer 的 Handover 平滑接管，再到 RUN 阶段速度环接管和故障保护。

- **当前目标**：让电机能够稳定完成 OFFSET_CALIB -> ALIGN -> OPENLOOP -> HANDOVER(blend) -> RUN 的启动流程，并在 RUN 阶段保持可观测、可调参、可保护。
- **工程目标**：形成清晰的软件分层和模块职责，避免所有逻辑堆在 main.c 或单个 foc.c 中，便于后续重构、团队交接和问题定位。
- **产品目标**：最终演进成“参数可标定、故障可诊断、控制可复现、固件可升级、测试可验证”的电机控制固件。
- **当前边界**：尚未覆盖完整 Flash 参数系统、Bootloader、正式上位机协议、硬件比较器全链路、温度模型、EMC 验证、长时间可靠性测试和量产标定流程。

系统边界需要特别明确：App 层负责组织流程，Control 层负责控制算法，BSP 层负责硬件访问，调试系统只读取快照。任何模块越界都会导致后续调试困难。例如采样模块如果开始判断启动状态，或者 PWM 模块开始判断 Observer 是否锁定，系统会迅速变成网状依赖，难以维护。

## 02_软件分层架构

当前工程已经形成了比较清晰的分层结构：系统启动层、Config 参数层、BSP 外设层、Control 控制层、App 应用编排层和 Diagnostics 诊断层。产品级工程建议继续保持这种方向，但要进一步强化接口边界和数据流方向。

- **系统启动层 / Core**：主要包括 main.c、system_stm32g4xx.c、stm32g4xx_it.c、HAL MSP 初始化文件。它只负责系统装配、中断入口和主循环调度，不应承载电机算法。
- **BSP 外设层 / BSP**：主要包括 bsp_adc、bsp_tim、bsp_usart、bsp_dac、bsp_gpio、bsp_dma、bsp_key、bsp_cordic。它负责把 HAL 外设配置封装成稳定接口，避免上层到处直接访问寄存器或 HAL 句柄。
- **控制算法层 / MotorControl**：Keil 中聚合为 MotorControl 分组，主要包括 foc_current_sense、foc_svpwm、foc_current_loop、foc_speed_loop、foc_startup、foc_observer、foc_handover、foc_deadtime_comp、foc_protection、foc_math、foc_types。它应尽量保持控制数据结构和控制计算，不直接做业务调度。
- **应用编排层 / App**：主要包括 app_foc、app_foc_timing 和 app_debug。app_foc 是系统控制中心，负责状态机、高频/中频任务编排、启动停机、速度环入口、故障快照；app_foc_timing 负责 DWT 高频耗时统计；app_debug 负责低频调试输出。
- **模型算法层 / MotorModel**：Keil 中聚合为 MotorModel 分组，主要包括 foc_clarke_park、foc_pi_controller、foc_flux_observer_pll。这些模块尽量保持纯算法特征，不直接访问 App 状态机和外设。
- **诊断工具层 / Diagnostics**：当前主要是 foc_trace、RuntimeSnapshot、TraceSnapshot、FaultSnapshot、HfTimingSnapshot 和 UART/DAC 调试。产品级应扩展为上位机协议、故障记录、数据回放和工厂测试接口。

```text
main.c / 中断入口
  -> AppFoc_Init / AppFoc_Start / AppFoc_MediumFreqTask / AppFoc_HighFreqTask
      -> FOC_CurrentSense
      -> FOC_Startup
      -> FOC_Observer
      -> FOC_Handover
      -> FOC_CurrentLoop
      -> FOC_Svpwm
      -> FOC_Protection
  -> AppDebug_Task 只读取快照，不反向修改控制对象
```

## 03_核心模块职责划分

核心模块的划分原则是：一个模块只解决一类问题，并且只暴露稳定接口。这样在电机抖动、无法切换、速度跟不上、采样异常、ISR 超时等问题出现时，才能快速判断问题属于采样链、角度链、电流环、调制链、状态机还是保护链。

### AppFoc

- **核心职责**：系统状态机、高频/中频任务编排、启动/停止、运行快照、故障快照、速度环入口、命令提交。
- **不应该承担**：不直接展开 Clarke/Park、PI、PLL、SVPWM 细节，不直接在多个地方操作 TIM/ADC 寄存器。

### FOC_CurrentSense

- **核心职责**：ADC 原始值读取、零偏校准、电流换算、母线电压换算、可信两相选择、第三相重构、fallback/hold-last 诊断。
- **不应该承担**：不判断是否进入 RUN，不决定电流给定，不直接改变 PWM 输出。

### FOC_Svpwm

- **核心职责**：SVPWM 扇区计算、T1/T2/T0、三相 CCR、采样窗口分析、ADC 触发 CCR4 计算。
- **不应该承担**：不读取 ADC，不做过流保护，不判断 Observer 锁定，不决定启动状态。

### FOC_CurrentLoop

- **核心职责**：Clarke/Park、Id/Iq PI、Iq-only、DSoft、前馈/解耦、dq 限幅、InvPark、输出 alpha/beta 电压指令。
- **不应该承担**：不推进主状态机，不直接开关驱动器，不决定故障恢复。

### FOC_Startup

- **核心职责**：ALIGN/OPENLOOP 的角度轨迹、机械频率爬坡、电角速度输出。
- **不应该承担**：不做闭环速度控制，不直接判断 Observer 是否可信。

### FOC_Observer

- **核心职责**：封装 foc_flux_observer_pll，输出估算角度、速度、磁链、PLL 状态、锁定状态。
- **不应该承担**：不决定是否接管控制角，不直接输出 PWM。

### FOC_Handover

- **核心职责**：判断 open angle 与 observer angle 是否满足接管条件，完成 hold、blend、observer 接管和异常回退。
- **不应该承担**：不计算 PLL，不运行电流环，不进行复杂故障恢复。

### FOC_Protection

- **核心职责**：软件故障检查、故障位图、停机动作接口、故障状态收口。
- **不应该承担**：不负责启动策略，不在内部进行复杂自动重启。

### AppDebug

- **核心职责**：低频读取 RuntimeSnapshot/HfTimingSnapshot/FaultSnapshot，并按策略打印或输出给上位机。
- **不应该承担**：不在高频中断中阻塞打印，不直接修改控制参数。

## 04_模块调用关系

模块调用关系按时间尺度分为四条链：启动初始化链、高频控制链、中频管理链、调试读取链。商业工程中最重要的是保证高频控制链短、固定、可测量；中频管理链可以做状态推进和速度环；调试读取链只能读取快照，不能反向污染控制状态。

- **启动初始化链**：main() 调用 HAL_Init、SystemClock_Config、MX_GPIO/MX_DMA/MX_USART/MX_ADC/MX_TIM/MX_CORDIC 等外设初始化，然后进入 AppFoc_Init 和 AppFoc_Start。
- **高频控制链**：TIM1 同步触发 ADC 注入采样，ADC2 注入完成回调进入 AppFoc_HighFreqTask，按当前状态执行 OFFSET_CALIB、ALIGN、OPENLOOP、RUN、FAULT 等对应链路。
- **中频管理链**：main while 周期调用 BSP_KEY_Task、AppFoc_MediumFreqTask、AppDebug_Task。AppFoc_MediumFreqTask 负责 1ms 状态机、速度环、命令提交和快照刷新。
- **调试读取链**：AppDebug_Task 通过 AppFoc_GetRuntimeSnapshot、AppFoc_GetHfTimingSnapshot、AppFoc_GetFaultSnapshot 获取稳定快照，并输出到 UART 或后续上位机协议。

```text
硬件采样完成
  -> ADC2 injected complete callback
    -> AppFoc_HighFreqTask()
      -> app_foc_current_sense_update()
      -> FOC_ProtectionCheckFault()
      -> 根据状态运行 ALIGN / OPENLOOP / RUN
      -> FOC_CurrentLoop...
      -> app_foc_apply_voltage_command()
      -> FOC_Svpwm_ApplySvpwm()
```

## 05_数据流总览

数据流应尽量保持单向：ADC 原始量进入采样模块，采样模块输出物理量，控制模块输出电压指令，PWM 模块写入 CCR，诊断模块只读取快照。这个原则能明显降低系统耦合度。

- **采样数据流**：ADC raw -> voltage/current offset -> phase current iu/iv/iw -> Clarke/Park -> id/iq。
- **控制数据流**：id_ref/iq_ref + id/iq + theta_ctrl -> PI + feedforward + limit -> vd/vq -> InvPark -> valpha/vbeta。
- **观测器数据流**：valpha/vbeta + ialpha/ibeta + vbus -> foc_flux_observer_pll -> theta_obs/speed_obs/flux/pll_err/locked。
- **切换数据流**：theta_open + theta_obs + open_speed + obs_speed + angle_err -> Handover -> theta_ctrl/blend_k/fail_reason。
- **PWM 数据流**：valpha/vbeta + vbus -> SVPWM sector/T1/T2/T0 -> CCR1/2/3 + ADC trigger CCR4。
- **诊断数据流**：各模块内部状态 -> RuntimeSnapshot/TraceSnapshot/FaultSnapshot -> UART/DAC/上位机/日志回放。

# 02_硬件与控制时序架构

本章重点说明“硬件资源如何服务于控制时序”。对于 PMSM FOC 工程来说，硬件时序不是附属内容，而是控制系统能否稳定运行的基础。电流采样点、PWM 更新点、ADC 触发源、中断入口、DWT 实时性余量、硬件保护链路，都会直接影响电流环稳定性、Observer 估算质量和无感切换成功率。

当前工程的硬件与时序主线可以概括为：

```text
TIM1 中心对齐三相互补 PWM
  -> TIM1_CH4 OC4REF 作为 TRGO
    -> ADC2 注入组同步采样 U/V/W 三相电流
      -> ADC2 注入完成中断
        -> AppFoc_HighFreqTask 高频 FOC 控制
          -> 更新电流环 / Observer / SVPWM / ADC 下一拍触发点
```

本章按照当前代码实现写，同时向商业工程落地补充必要内容。当前代码中比较关键的实现依据包括：`foc_config.h`、`bsp_tim.c`、`bsp_adc.c`、`main.h`、`stm32g4xx_it.c`、`foc_svpwm.c`、`foc_current_sense.c`。后续如果改动 PWM 频率、ADC 通道、TIM1 TRGO、采样窗口、死区时间、中断优先级，必须同步更新本章。

## 01_硬件平台组成

当前硬件平台可以抽象成“STM32G474 + 三相功率逆变桥 + 三低边电流采样 + 母线电压采样 + 调试接口”的结构。它不是单纯的软件 FOC，而是软硬件强耦合的实时控制系统。

### 当前硬件组成

当前工程按以下硬件资源组织：

- **主控 MCU**：STM32G474，负责 PWM 输出、ADC 采样、FOC 算法、状态机、调试通信和保护管理。
- **功率级**：三相逆变桥，上桥和下桥由 TIM1 的互补 PWM 控制。
- **电机类型**：低压 PMSM，当前配置按 24V 母线、4 极对数、三相 FOC 控制设计。
- **电流采样方式**：三低边采样，U/V/W 三相电流分别通过采样电阻和运放送入 ADC。
- **母线电压采样**：母线电压通过电阻分压后进入 ADC1 常规通道。
- **驱动使能**：M1_EN_DRIVER GPIO 控制功率驱动器使能。
- **调试输出**：UART 用于串口打印，DAC1/DAC2 用于波形观测预留。
- **按键输入**：KEY 和 Start/Stop 外部中断用于本地启动、停止或调试命令。

### 当前代码中的主要引脚关系

当前 `main.h` 中定义的关键引脚如下：

```text
三相电流采样：
  U 相电流：PC1 -> ADC12_IN7
  V 相电流：PC2 -> ADC12_IN8
  W 相电流：PC3 -> ADC12_IN9

母线电压采样：
  VBUS：PA3 -> ADC1_IN4

三相 PWM 高边：
  U 高边：PA8  -> TIM1_CH1
  V 高边：PA9  -> TIM1_CH2
  W 高边：PA10 -> TIM1_CH3

三相 PWM 低边：
  U 低边：PB13 -> TIM1_CH1N
  V 低边：PB14 -> TIM1_CH2N
  W 低边：PB15 -> TIM1_CH3N

驱动与保护：
  M1_EN_DRIVER：PC13
  M1_BRAKE：PB12

调试：
  DAC_CH1：PA4
  DAC_CH2：PA6
  Start/Stop：PE14 -> EXTI15_10
```

这些引脚关系对控制时序有直接影响。例如 U/V/W 电流注入通道的 rank 顺序必须和 `foc_current_sense.c` 中的读取顺序一致；三相 PWM 的 U/V/W 映射必须和 SVPWM 输出的 ccr_u/ccr_v/ccr_w 一致；驱动使能脚必须只由 App 层统一管理，避免 Control 层私自开关功率输出。

### 当前硬件参数与采样换算

当前工程中和硬件采样相关的核心参数是：

```text
ADC 基准电压：FOC_ADC_VREF = 3.3 V
ADC 分辨率：FOC_ADC_FULL_SCALE = 4096
电流采样电阻：FOC_SHUNT_OHM = 0.02 Ω
电流采样放大倍数：FOC_GAIN = 6.0
母线电压分压比例：FOC_VBUS_RATIO = 25.0
软件电流保护限值：FOC_CURRENT_LIMIT_A = 15.0 A
母线欠压阈值：FOC_VBUS_UNDERVOLTAGE_V = 10.0 V
母线过压阈值：FOC_VBUS_OVERVOLTAGE_V = 30.0 V
```

由这些参数可以得到几个关键工程量：

```text
ADC 每 count 电压：
  V_lsb = 3.3 / 4096 = 0.0008057 V/count

电流采样每 count 对应电流：
  I_lsb = V_lsb / (Rshunt * Gain)
        = 0.0008057 / (0.02 * 6)
        = 0.006714 A/count
        ≈ 6.7 mA/count

母线电压每 count 对应实际电压：
  Vbus_lsb = V_lsb * 25
           = 0.02014 V/count
           ≈ 20.1 mV/count

24V 母线对应 ADC 计数：
  raw_vbus = 24 / 25 / 3.3 * 4096
           ≈ 1192 count
```

电流采样的量程也要重点关注。以当前 0.02Ω 采样电阻和 6 倍放大为例：

```text
1A 相电流对应运放输出变化：
  ΔVadc = 1 * 0.02 * 6 = 0.12 V
  Δcount = 0.12 / 3.3 * 4096 ≈ 149 count

15A 相电流对应运放输出变化：
  ΔVadc = 15 * 0.02 * 6 = 1.8 V
  Δcount = 1.8 / 3.3 * 4096 ≈ 2234 count
```

如果电流采样零点在 1.65V 附近，那么正负方向的可用摆幅约为 ±1.65V。15A 对应 1.8V，已经接近甚至超过单方向理想摆幅。因此当前 `FOC_CURRENT_LIMIT_A = 15A` 更适合作为研发阶段上限值，产品级需要结合运放输出范围、ADC 饱和、实际零偏、噪声裕量重新评估。

### 当前硬件平台的工程定位

当前硬件平台适合完成：

- 三相电流闭环验证；
- 无感 Observer/PLL 验证；
- Startup/OpenLoop 到 Observer 的 Handover 接管调试；
- 速度环和前馈补偿实验；
- DWT 实时性评估；
- 串口/DAC 调试。

但距离商业成品还需要补齐：

- 驱动器 FAULT 到 MCU 的完整链路；
- TIM1 BKIN 硬件保护；
- 温度采样与过温保护；
- 电流采样饱和检测；
- 母线电压标定；
- 过流阈值硬件验证；
- EMI/ESD/浪涌保护；
- 量产测试点和工装接口。

## 02_MCU外设资源分配

外设分配决定了整个控制系统的实时边界。FOC 工程中，TIM1、ADC、GPIO、UART、DAC 不能随便交叉使用，否则很容易出现“能编译、能运行，但时序不稳定”的问题。

### 当前外设归属

当前工程建议按以下方式理解外设职责：

- **TIM1**：三相中心对齐互补 PWM 输出，同时通过 CH4 OC4REF 产生 ADC 注入触发。
- **ADC2 注入组**：U/V/W 三相电流采样，是高频控制链路的入口。
- **ADC1 常规组**：母线电压采样，低频或后台读取。
- **ADC1 注入组**：当前代码中也配置了 PC1/PC2 注入采样，主要用于原始值对照调试，不作为主电流反馈链路。
- **USART1**：串口打印、后续上位机协议预留。
- **DAC1/DAC2**：调试波形输出预留，用于观察电流、角度、速度、误差等模拟量。
- **GPIO**：驱动使能、按键、Brake 输入、LED 或调试翻转。
- **CORDIC**：三角函数或数学加速预留，当前控制算法可逐步接入。
- **DMA**：当前主要服务外设基础配置，产品级可用于高速采样缓存或通信接收。

### 系统时钟与时序基础

当前 `SystemClock_Config()` 使用 HSE + PLL。按工程中 `stm32g4xx_hal_conf.h` 的 HSE_VALUE = 8MHz，以及 PLLM=2、PLLN=85、PLLR=2 推算：

```text
PLL 输入频率：
  8 MHz / 2 = 4 MHz

PLL VCO：
  4 MHz * 85 = 340 MHz

SYSCLK：
  340 MHz / 2 = 170 MHz

HCLK / PCLK1 / PCLK2：
  均为 170 MHz
```

这与当前 PWM 参数能够对应起来：TIM1 Prescaler=0、ARR=5311、中心对齐后得到约 16kHz PWM/控制周期。

需要注意一个潜在维护点：当前工程中 `system_stm32g4xx.c` 里也出现过 HSE_VALUE = 24000000U 的默认定义，而 `stm32g4xx_hal_conf.h` 中是 8000000UL。实际编译时哪个定义生效，要看工程包含顺序和编译宏。产品级必须统一 HSE_VALUE，否则以下内容可能出错：

- HAL 计算的 SystemCoreClock；
- DWT cycles 到 us 的换算；
- UART 波特率计算；
- ADC/TIM 派生时钟认知；
- 文档中的实时性计算。

本项目从 TIM1 ARR 和 `FOC_TS_SEC = 62.5us` 反推，当前设计目标显然是 170MHz 系统时钟和约 16kHz FOC 控制周期。因此建议把 HSE_VALUE 统一到实际晶振值，并在 README/ARCHITECTURE 中明确写死。

### 中断优先级分配

当前代码中与控制时序相关的 NVIC 设置包括：

```text
ADC1_2_IRQn：
  Priority = 2, SubPriority = 0
  用途：ADC2 注入完成后进入 AppFoc_HighFreqTask

TIM1_BRK_TIM15_IRQn：
  Priority = 4, SubPriority = 1
  用途：TIM1 Break / TIM15 中断

TIM1_UP_TIM16_IRQn：
  Priority = 0, SubPriority = 0
  用途：TIM1 Update / TIM16 中断
  当前注释说明：FOC 高频任务不是由 TIM1 Update 调度，而是由 ADC2 注入完成回调调度
```

这里有一个产品级需要注意的点：当前 TIM1 Update 中断优先级为 0，比 ADC1_2 更高，但它并不是 FOC 调度入口。如果 TIM1 Update 中断里没有实际控制任务，建议后续评估是否降低优先级或关闭，避免它无意义地抢占 ADC2 高频控制链路。当前真正的高频控制入口是 ADC2 注入完成中断，因此 ADC1_2_IRQn 的稳定性最重要。

同时，TIM1 Break 的软件中断优先级当前低于 ADC1_2。若产品级启用硬件 Break，硬件关断动作本身不依赖中断优先级；Break ISR 主要负责记录故障和进入 FAULT。但如果 BreakState 仍然关闭，那么单靠软件中断不能替代硬件保护。

### 外设分配原则

后续维护建议遵守以下原则：

- TIM1 不要再混入非电机控制用途。
- ADC2 注入组只服务三相电流采样，不要加入慢速温度、电位器等通道。
- 母线电压、温度等慢变量优先放 ADC1 常规组。
- UART 打印不能进入高频中断。
- DAC 输出只能做轻量级调试，不能在高频中做复杂格式化。
- KEY/EXTI 只产生命令请求，不直接在中断里启动/停止电机。
- CORDIC 或数学库替换必须重新测 DWT 耗时。
- 任何外设初始化变更都必须同步检查 Keil 工程文件和中断入口。

## 03_TIM1_PWM时序

TIM1 是当前系统最核心的时基。它不仅输出三相 PWM，还通过 CH4 产生 ADC 触发点。因此 TIM1 的计数模式、ARR、CCR、死区、TRGO 都会影响整个 FOC 链路。

### 当前 TIM1 配置

当前 `bsp_tim.c` 中 TIM1 的关键配置为：

```text
Prescaler = 0
CounterMode = TIM_COUNTERMODE_CENTERALIGNED1
Period = 5311
ClockDivision = TIM_CLOCKDIVISION_DIV2
RepetitionCounter = 1
AutoReloadPreload = DISABLE

CH1/CH2/CH3：
  PWM1 模式
  初始 Pulse = 2655
  用于 U/V/W 三相高边 PWM

CH1N/CH2N/CH3N：
  互补低边 PWM

CH4：
  PWM2 模式
  初始 Pulse = 2655
  用于 OC4REF -> TIM1 TRGO -> ADC 注入触发

TRGO：
  MasterOutputTrigger = TIM_TRGO_OC4REF

DeadTime：
  DeadTime = 85

Break：
  BreakState = DISABLE
  Break2State = DISABLE
```

这说明当前工程不是用 TIM1 Update 直接触发 FOC，而是用 TIM1 CH4 的 OC4REF 作为 ADC 触发源，ADC2 注入采样完成后再进入高频控制。

### PWM 频率计算

按 170MHz TIM1 计数时钟估算，TIM1 每个计数 tick 为：

```text
T_tick = 1 / 170MHz = 5.882 ns
```

中心对齐模式下，一个完整 PWM 周期包含上数和下数两个过程：

```text
T_pwm = 2 * (ARR + 1) * T_tick
      = 2 * 5312 / 170000000
      = 62.494 us

f_pwm = 1 / T_pwm
      ≈ 16.002 kHz
```

这与 `FOC_TS_SEC = 6.25e-5s` 基本一致。当前代码中 `FOC_TS_SEC` 就应该理解为高频 FOC 控制周期，也就是约 62.5us。

### PWM 分辨率

当前 ARR=5311，因此比较值的有效分辨率约为：

```text
每 1 个 CCR count 对应 duty：
  1 / 5311 = 0.0188%

每 100 个 CCR count 对应 duty：
  100 / 5311 = 1.88%
```

这说明当前 PWM 分辨率比较充足。对 24V 母线而言，粗略估算 1 个 count 对应的母线占空比电压分辨率为：

```text
24V * 1 / 5311 ≈ 4.5 mV
```

实际输出到相电压还会受到 SVPWM、死区、母线波动、MOSFET 压降影响，不能简单等同，但这个值能帮助判断 PWM 量化误差不是当前主要矛盾。

### 中点输出含义

当前 `FOC_PWM_HALF_ARR = 2655`，三相 CCR 都等于 2655 时，等效三相 50% duty。对于三相桥来说，这不是“完全没有开关动作”，而是三相都输出同样的平均电压，因此电机线电压近似为 0。

当前工程中以下场景会使用中点输出：

- IDLE 安全状态；
- OFFSET_CALIB 零偏校准；
- FAULT 安全收口；
- 初始化阶段 PWM 未真正输出有效电压前；
- ADC 触发 fallback 时的保守采样点。

中点输出的价值是：保持 PWM 定时链路存在，同时让三相线电压为零，便于校准、停机和安全收口。

### 死区时间计算

当前 TIM1 配置：

```text
ClockDivision = TIM_CLOCKDIVISION_DIV2
DeadTime = 85
```

在 STM32 高级定时器中，ClockDivision 会影响 tDTS。按 170MHz 计数时钟估算：

```text
t_CK_INT = 1 / 170MHz = 5.882 ns
t_DTS = 2 * t_CK_INT = 11.765 ns

DeadTime ≈ 85 * 11.765 ns
         ≈ 1.000 us
```

这个结果和当前 `FOC_DT_COMP_K_BASE = 0.016` 是对应的：

```text
死区占控制周期比例：
  K = Td / Ts
    = 1.0us / 62.5us
    = 0.016
```

如果母线电压为 24V，死区引起的等效电压误差量级可以粗略估算为：

```text
V_dt ≈ Vbus * Td / Ts
     ≈ 24 * 0.016
     ≈ 0.384 V
```

这个电压误差在低速、小电流时会表现为电流畸变、过零不平滑、噪声增加；在高速时会表现为电压余量被吃掉。当前死区补偿已经预留，但 ALIGN/OPENLOOP/RUN 阶段默认仍关闭，说明当前阶段主要还是先保证主控制链稳定。

### TIM1 时序维护原则

后续不要随意修改以下配置：

- CenterAligned 模式；
- ARR；
- CH4 作为 ADC 触发；
- 死区时间；
- PWM 极性；
- 互补输出极性；
- OffStateRun/Idle；
- Break 配置；
- RepetitionCounter。

如果必须修改，至少要重新验证：

- 实际 PWM 频率；
- ADC 触发点是否仍在低边稳定导通窗口；
- ADC 注入完成中断频率；
- DWT 测得高频任务周期；
- 三相 PWM 是否存在毛刺；
- 中点输出是否安全；
- 停机时三相是否回到预期状态。

## 04_ADC采样时序

ADC 采样时序是电流环的根。如果采样点不对，后面的 Clarke/Park、PI、Observer、Handover 全都会被污染。当前工程已经从“固定中点采样”向“根据 SVPWM 扇区动态移动采样点”演进，这是比较接近商业工程的方向。

### 当前 ADC 资源使用

当前 `bsp_adc.c` 的主要配置如下：

```text
ADC1 常规组：
  通道：ADC_CHANNEL_4
  引脚：PA3
  功能：母线电压 VBUS
  模式：ContinuousConvMode = ENABLE
  触发：Software start
  采样时间：ADC_SAMPLETIME_640CYCLES_5

ADC1 注入组：
  通道：ADC_CHANNEL_7 / ADC_CHANNEL_8
  引脚：PC1 / PC2
  功能：U/V 电流原始值对照调试
  触发：TIM1_TRGO
  采样时间：ADC_SAMPLETIME_12CYCLES_5

ADC2 注入组：
  通道：ADC_CHANNEL_7 / ADC_CHANNEL_8 / ADC_CHANNEL_9
  引脚：PC1 / PC2 / PC3
  功能：U/V/W 三相电流主采样
  触发：TIM1_TRGO
  触发边沿：RISING
  采样时间：ADC_SAMPLETIME_12CYCLES_5
  注入转换数：3
```

当前高频控制只使用 hadc2 注入完成回调作为 FOC 调度入口。ADC1 注入组即便配置了，也不应该参与主电流计算，主要用于对比调试。

### ADC 转换时间估算

当前 ADC ClockPrescaler 为 `ADC_CLOCK_SYNC_PCLK_DIV4`。按 170MHz PCLK 估算，ADC 时钟约为：

```text
F_adc ≈ 170MHz / 4 = 42.5MHz
T_adc = 1 / 42.5MHz = 23.53 ns
```

对于 ADC2 电流注入采样，采样时间为 12.5 cycles。12 位 ADC 通常还需要约 12.5 cycles 的转换时间，因此单通道时间可估算为：

```text
T_one_channel ≈ (12.5 + 12.5) / 42.5MHz
              ≈ 25 / 42.5MHz
              ≈ 0.588 us
```

三相注入顺序转换时间为：

```text
T_three_channel ≈ 3 * 0.588us
                ≈ 1.765 us
```

即使加上注入序列、采样保持、HAL 回调等开销，ADC 转换本身相对于 62.5us 控制周期仍然较小。真正需要关注的是采样点是否落在正确窗口，而不是 ADC 转换时间是否够用。

对于 ADC1 母线电压常规采样，采样时间为 640.5 cycles：

```text
T_vbus ≈ (640.5 + 12.5) / 42.5MHz
       ≈ 653 / 42.5MHz
       ≈ 15.36 us
```

母线电压不是每个高频周期都必须精准更新的高速量，因此给较长采样时间是合理的。较长采样时间可以降低母线分压电阻较大时的采样误差。

### 电流采样换算链路

当前 `foc_current_sense.c` 中的换算链路是：

```text
ADC raw
  -> ADC 引脚电压 Vadc
    -> 减去 offset 电压
      -> 除以 Gain * Rshunt
        -> 乘以相电流方向 sign
          -> 得到 iu/iv/iw
```

公式为：

```text
Vadc = raw * Vref / 4096

Iphase = sign * (Vadc - Voffset) / (Gain * Rshunt)
```

代入当前参数：

```text
Gain * Rshunt = 6 * 0.02 = 0.12 V/A

Iphase = (Vadc - Voffset) / 0.12
```

这意味着：

- 0.12V 的 ADC 电压变化约等于 1A；
- 1 count 约等于 6.7mA；
- offset 漂移 10 count 约等于 67mA；
- offset 漂移 100 count 约等于 0.67A。

所以电流零偏不是小问题。如果 offset 校准不稳定，低速 id/iq 会直接偏移，Observer 也会被污染。

### 电流零偏校准时间

当前 `FOC_CURRENT_OFFSET_CALIB_SAMPLES = 4096`，若按每个高频周期采一次，校准时间约为：

```text
T_offset = 4096 * 62.5us
         = 256ms
```

当前 ALIGN 时间为 300ms，因此从启动到真正进入 OPENLOOP 之前，基础时间约为：

```text
OFFSET_CALIB ≈ 256ms
ALIGN        = 300ms
合计         ≈ 556ms
```

这个时间对于研发验证是合理的。产品级需要根据启动响应要求评估是否缩短 offset 采样数，或者使用上电标定 + 运行中慢速 offset 漂移补偿。

### 三低边采样与可信两相重构

三低边采样的关键问题是：在某些 PWM 状态下，并不是三相电流都能同时可信测量。当前采样模块根据 SVPWM 扇区选择两相可信电流，再用三相电流和为 0 重构第三相。

当前代码的扇区与可信相关系是：

```text
Sector 1 / 6：
  可信相：V / W
  重构相：U = -(V + W)

Sector 2 / 3：
  可信相：U / W
  重构相：V = -(U + W)

Sector 4 / 5：
  可信相：U / V
  重构相：W = -(U + V)
```

这个关系同时在 `foc_current_sense.c` 和 `foc_svpwm.c` 中体现。后续如果修改 SVPWM 扇区定义，必须同步修改可信相选择，否则会出现“PWM 扇区正确，但电流重构错误”的隐蔽故障。

### 采样异常与 hold-last

当前工程支持：

```text
FOC_ADC_HOLD_LAST_ON_FALLBACK = 1
FOC_ADC_FALLBACK_HOLD_MAX_TICKS = 80
FOC_ADC_FALLBACK_HOLD_FAULT_ENABLE = 0
```

意思是：当当前 PWM 窗口不足时，如果已经有上一拍有效电流，就沿用上一拍电流，避免把明显不可信的采样值送入电流环。连续 hold-last 超过 80 拍会记录诊断量，但默认不直接触发故障。

80 拍对应时间为：

```text
T_hold_max = 80 * 62.5us = 5.0ms
```

研发阶段先记录不故障是合理的，因为这样有利于观察低占空比时采样窗口不足的频率。但产品级不能长期静默 hold-last，否则电流环会基于旧反馈继续输出电压。后续建议把连续 fallback/hold-last 纳入采样异常保护。

## 05_高频中断控制时序

高频中断是整个 FOC 控制系统的实时核心。当前工程不是由 `while(1)` 调用高频控制，也不是由 TIM1 Update 直接调用高频控制，而是由 ADC2 注入完成回调调用 `AppFoc_HighFreqTask()`。

### 当前高频入口

当前 `stm32g4xx_it.c` 中高频入口为：

```text
ADC1_2_IRQHandler()
  -> HAL_ADC_IRQHandler(&hadc2)
    -> HAL_ADCEx_InjectedConvCpltCallback(hadc)
      -> if (hadc == &hadc2)
           AppFoc_HighFreqTask()
```

这种设计的好处是：FOC 计算发生在 ADC 注入采样完成之后，高频任务读取到的是本拍刚刚转换完成的电流数据，而不是上一拍或不确定时刻的数据。

### 高频任务应该完成的动作

高频任务在不同状态下做的事情不同，但整体上应保持短、固定、无阻塞。

在 **OFFSET_CALIB** 状态：

- PWM 保持中点；
- 驱动保持关闭或安全状态；
- 采样模块累计 U/V/W offset；
- 不运行电流环；
- 不运行 Observer 接管；
- 不输出有效转矩。

在 **ALIGN** 状态：

- 使用固定电角度；
- 给定 d 轴对齐电流；
- q 轴电流为 0；
- 运行电流环；
- 输出 SVPWM；
- 保持一段时间让转子吸稳。

在 **OPENLOOP** 状态：

- 推进开环机械频率和电角度；
- 采集三相电流；
- 运行电流环或开环电压模式；
- 后台更新 Observer；
- 更新 Handover 判据；
- 根据 blend 状态选择控制角；
- 输出 SVPWM 和下一拍 ADC 触发点。

在 **RUN** 状态：

- 使用 Observer 角度作为控制角；
- 使用速度环给出的 iq_ref；
- 运行 Id/Iq 电流环；
- 可叠加前馈/解耦补偿；
- 更新 SVPWM；
- 更新保护和快照。

在 **FAULT** 状态：

- PWM 收口；
- 驱动关闭；
- 不再推进控制器积分；
- 保留故障快照。

### 高频周期预算

当前控制周期为 62.5us。按 170MHz CPU 频率换算：

```text
一个控制周期 CPU cycles：
  cycles_per_period = 170MHz * 62.5us
                    = 10625 cycles
```

如果 DWT 阈值设置为 8500 cycles，则对应时间为：

```text
T_threshold = 8500 / 170MHz
            = 50.0us
```

这意味着高频任务超过 50us 就进入 overrun 统计，距离 62.5us 的硬周期还剩约 12.5us 裕量。

如果实际高频耗时为 38us：

```text
CPU 占用比例 = 38 / 62.5 = 60.8%
剩余时间     = 24.5us
```

如果实际高频耗时为 40us：

```text
CPU 占用比例 = 40 / 62.5 = 64.0%
剩余时间     = 22.5us
```

这个余量在 16kHz 下可以接受，但不是“无限余量”。后续加入更多功能时必须重新测量，包括：

- 前馈/解耦补偿；
- 死区补偿；
- 动态采样窗口计算；
- 更多保护判断；
- TraceSnapshot 高频更新；
- CORDIC 或软件 sin/cos 替换；
- Observer 参数升级；
- 异常分支和故障快照。

### 如果提高 PWM 频率的影响

提高 PWM 频率会直接缩短高频任务预算。假设高频任务耗时仍为 40us：

```text
16kHz：
  Ts = 62.5us
  CPU 占用 ≈ 64%

20kHz：
  Ts = 50us
  CPU 占用 ≈ 80%

24kHz：
  Ts = 41.7us
  CPU 占用 ≈ 96%

25kHz：
  Ts = 40us
  CPU 占用 ≈ 100%
```

所以当前如果考虑提高 PWM 频率，建议先不要直接跳到 24kHz 或 25kHz。更稳妥的步骤是：

- 先关闭不必要的高频调试快照；
- 确认最坏路径耗时，而不是平均耗时；
- 先试 18kHz 或 20kHz；
- 每次改频率后重新确认 ADC 触发窗口；
- 重新确认电流环 PI 参数是否需要按 Ts 调整；
- 重新确认死区补偿 K=Td/Ts；
- 重新确认 DWT overrun 阈值。

### 高频中断禁止事项

高频中断中禁止或不建议做：

- printf；
- UART 阻塞发送；
- Flash 写入；
- malloc/free；
- 复杂字符串处理；
- 上位机协议解析；
- 长循环搜索；
- 等待标志位；
- HAL_Delay；
- 文件/日志持久化；
- 大量浮点打印格式化。

这些动作应放在中频或低频任务中，通过快照读取高频数据。

## 06_中频任务调度时序

中频任务负责“慢决策”，高频任务负责“快执行”。当前 `main.c` 的主循环中按顺序调用：

```text
while (1)
{
    BSP_KEY_Task();
    AppFoc_MediumFreqTask();
    AppDebug_Task();
}
```

这说明中频任务不是硬件中断强实时调度，而是在主循环中运行。它的实时性弱于 ADC 高频中断，但足够承担状态机推进、速度环、调试打印和命令处理。

### 中频任务的定位

中频任务适合做：

- 启动/停止命令处理；
- 状态机推进；
- 速度环计算；
- 速度目标斜坡；
- RUN 进入后延迟使能速度环；
- 快照刷新；
- 按键消抖；
- 调试打印节拍控制；
- 低频保护诊断；
- 上位机命令解析。

中频任务不应该直接做：

- 三相电流采样；
- PWM CCR 高频更新；
- Park/InvPark 高频计算；
- 每拍 Observer 更新；
- 每拍 SVPWM 计算；
- 高频故障快速关断。

### 高频与中频的时间比例

当前高频周期约 62.5us，即 16kHz。若中频按 1ms 运行，则：

```text
1ms 内包含的高频周期数：
  1ms / 62.5us = 16
```

也就是说，每 16 次电流环周期，速度环或状态机大约更新一次。这是常见的内外环比例：

```text
电流环：16kHz
速度环：约 1kHz
调试打印：几十 Hz 或更低
参数保存：极低频，不能在控制周期内做
```

当前速度环参数中有：

```text
FOC_SPEED_ENABLE_DELAY_MS = 500
FOC_SPEED_TAKEOVER_HOLD_MS = 300
FOC_SPEED_REF_RAMP_HZ_PER_S = 3.0
FOC_SPEED_IQ_SLEW_A_PER_S = 5.0
```

这些参数本质上都是中频时间尺度。比如速度给定斜率 3Hz/s，如果 1ms 更新一次，则每次更新目标机械频率约：

```text
Δf_per_ms = 3Hz/s * 0.001s = 0.003Hz
```

Iq 斜率 5A/s，如果 1ms 更新一次，则每次 iq_ref 最大变化约：

```text
ΔIq_per_ms = 5A/s * 0.001s = 0.005A
```

这些斜率可以避免 RUN 进入后速度环突然给很大的 iq_ref，引起电流冲击或 Observer 失锁。

### 中频任务的产品级改进

当前 while(1) 轮询式中频任务适合研发阶段。产品级可以继续保留这种结构，但建议补充：

- 统一 soft timer；
- 明确 1ms、10ms、100ms、1s 任务表；
- 每个任务记录最大执行时间；
- AppDebug 打印限速；
- 上位机通信解析限时；
- 参数保存延迟提交；
- 故障状态下禁止执行危险命令；
- 正常停机和故障停机分离。

建议的产品级中频节拍如下：

```text
1ms：
  状态机推进
  速度环
  命令提交
  快照刷新

10ms：
  按键处理
  通信心跳
  慢速保护诊断
  温度滤波

100ms：
  运行统计
  打印摘要
  上位机状态帧
  参数脏标志检查

1s：
  运行时间累计
  最大/最小值统计
  健康状态更新
  故障历史维护
```

## 07_PWM_ADC同步关系

PWM 与 ADC 的同步关系是三低边采样架构的核心。当前工程已经把 ADC 触发点计算放入 `foc_svpwm.c`，这是一个很重要的设计：PWM 模块不仅负责输出电压，还负责告诉 ADC “应该在哪里采样”。

### 当前同步链路

当前同步链路如下：

```text
FOC_Svpwm_ApplySvpwm()
  -> 计算 sector / T1 / T2 / T0
  -> 计算 ccr_u / ccr_v / ccr_w
  -> 写入 TIM1_CH1 / CH2 / CH3
  -> foc_svpwm_update_adc_trigger_ccr()
    -> 根据当前扇区选择可信两相
    -> 计算安全采样窗口
    -> 写入 TIM1_CH4
      -> TIM1_TRGO_OC4REF
        -> ADC2 注入组触发
```

这样每一拍 SVPWM 计算完成后，下一拍 ADC 触发点也随之更新。对于低边采样来说，这是比固定中点采样更可靠的结构。

### CH4 触发点的作用

当前 TIM1 CH4 使用 PWM2 模式，并作为 TRGO 输出。ADC2 注入组配置为：

```text
ExternalTrigInjecConv = ADC_EXTERNALTRIGINJEC_T1_TRGO
ExternalTrigInjecConvEdge = RISING
```

因此 ADC 注入采样不是随便开始的，而是由 TIM1_CH4 的 OC4REF 上升沿触发。`foc_svpwm_update_adc_trigger_ccr()` 最终会调用：

```text
__HAL_TIM_SET_COMPARE(&htim1, TIM_CHANNEL_4, pwm->adc_sample_ccr)
```

这意味着 `adc_sample_ccr` 是当前系统非常关键的时序变量。调试采样问题时，不应该只看 U/V/W raw，也要看：

- 当前 sector；
- ccr_u/ccr_v/ccr_w；
- trusted_pair；
- adc_sample_ccr；
- adc_win1_ccr；
- adc_win2_ccr；
- adc_trigger_fallback；
- midpoint_sampling；
- adc_dynamic_sampling。

### 安全采样窗口计算

当前代码的核心思路是：对三低边采样来说，可信两相的低边都导通后，才能采到可信电流。当前高边是 PWM1，低边互补输出可以近似理解为在 `CNT > CCR` 后进入稳定导通区。因此对某个扇区的可信两相，安全窗口近似为：

```text
safe_start = max(ccr_x, ccr_y) + margin
safe_end   = ARR - margin
safe_width = safe_end - safe_start
```

当前配置：

```text
FOC_ADC_TRIG_MARGIN_CCR = 140
FOC_ADC_TRIG_MIN_WINDOW_CCR = 320
FOC_ADC_SAMPLE_MIN_CCR = 140
FOC_ADC_SAMPLE_MAX_CCR = 5311 - 140 = 5171
```

换算成时间：

```text
1 CCR count = 5.882ns

margin 140 count：
  140 * 5.882ns = 0.824us

min window 320 count：
  320 * 5.882ns = 1.882us
```

这表示采样点距离开关边沿至少预留约 0.82us，且安全窗口宽度至少要有约 1.88us 才认为可用。这个预留用来避开：

- MOSFET 开关瞬态；
- 死区时间；
- 运放输出建立时间；
- ADC 采样保持时间；
- 电流尖峰和共模扰动。

### 为什么固定中点采样不总是可靠

固定中点采样的思路是把 ADC 触发点放在 `FOC_PWM_HALF_ARR = 2655`。这在占空比中等、可信两相 CCR 都小于中点时通常可以工作。

但是如果某个可信相的 CCR 已经超过中点，例如：

```text
ccr_x = 3100
ccr_y = 1800
center = 2655
```

则可信两相的公共低边导通窗口要从：

```text
safe_start = max(3100, 1800) + 140 = 3240
```

开始。此时中心点 2655 已经落在安全窗口之前，固定中点采样会采到不稳定电流。动态采样会把 sample 移到安全窗口中间，从而提高采样可信度。

### fallback 与 hold-last 的意义

如果安全窗口太窄：

```text
safe_width < FOC_ADC_TRIG_MIN_WINDOW_CCR
```

当前 PWM 模块会：

- 设置 `adc_trigger_fallback = 1`；
- 使用 `FOC_ADC_TRIG_FALLBACK_CCR`，也就是中点 2655；
- 通知采样模块 fallback_request；
- 若允许 hold-last，则采样模块沿用上一拍有效电流。

这不是理想控制状态，而是一种保护性降级。它的意义是避免把明显错误的电流送入 PI。但如果 fallback 长时间持续，说明当前 PWM 占空比、采样点策略或低边采样条件已经不满足，需要进入诊断或降级。

### 产品级同步验证方法

产品级不能只靠软件推导确认 PWM/ADC 同步，必须上示波器或逻辑分析仪验证：

- CH1/CH2/CH3 PWM 波形；
- CH4 触发位置；
- ADC 触发时刻与三相低边导通窗口关系；
- 电流采样运放输出在触发点附近是否稳定；
- 死区后多久电流信号恢复稳定；
- fallback 发生时的实际占空比和电流波形。

推荐调试打印字段：

```text
sec
pair
fb
ccru/ccrv/ccrw
win1/win2
smp
raw_u/raw_v/raw_w
iu/iv/iw
isum = iu + iv + iw
```

如果 `isum` 长期偏离 0，优先检查采样方向、可信相选择、ADC 通道映射和采样点，而不是先调 PI 或 Observer。

## 08_DWT执行时间与实时性评估

DWT 执行时间统计是判断工程是否还能继续加功能的重要依据。FOC 系统不能只看“能不能跑”，还要看“最坏情况下能不能准时跑完”。

### DWT 的基本原理

Cortex-M4 内核中 DWT CYCCNT 是一个 CPU cycle 计数器。高频任务开始时读取一次，结束时再读取一次，两者相减就是本次高频任务消耗的 CPU cycles。

换算公式为：

```text
执行时间 us = cycles / CPU_Hz * 1e6
```

按当前 170MHz CPU 频率：

```text
1 cycle = 1 / 170MHz = 5.882ns

1000 cycles = 5.882us
5000 cycles = 29.41us
8500 cycles = 50.00us
10625 cycles = 62.50us
```

因此，如果 DWT 打印中出现：

```text
hf_cycle = 6500
```

对应时间约为：

```text
6500 / 170MHz = 38.24us
```

### 实时性不只看平均值

实时性评估要看以下指标：

- last：最近一次耗时；
- min：最小耗时；
- max：最大耗时；
- avg：平均耗时；
- threshold：阈值；
- overrun_count：超过阈值次数；
- 故障分支耗时；
- RUN + 前馈 + 保护全开时耗时；
- 调试快照打开时耗时；
- 长时间运行最大值。

平均值低并不代表安全。如果 max 接近 62.5us，说明系统已经有偶发超时风险。电机控制里，偶发超时可能导致采样相位漂移、电流环延迟突变、Observer 相位误差增加，最终表现为噪声、抖动或失步。

### 当前工程的预算模型

当前周期为 62.5us。建议把预算分成几层：

```text
硬极限：
  62.5us
  超过这个值，高频控制已经跨周期

软件报警阈值：
  50us
  约 8500 cycles，超过就记录 overrun

建议长期运行目标：
  <= 40us
  约 64% CPU 占用

更理想的产品级目标：
  <= 35us
  约 56% CPU 占用
```

如果后续要提高 PWM 频率，必须先看当前最坏耗时。例如：

```text
当前最坏耗时 40us：

16kHz：周期 62.5us，剩余 22.5us
20kHz：周期 50.0us，剩余 10.0us
24kHz：周期 41.7us，剩余 1.7us
25kHz：周期 40.0us，几乎无剩余
```

所以当前工程即使在 16kHz 下还有余量，也不能直接判断 24kHz 一定安全。提高 PWM 频率之前必须先减少高频任务耗时，或者把部分诊断和计算移到中频。

### 哪些改动会增加高频耗时

后续这些改动都会增加高频耗时：

- 每拍更新完整 TraceSnapshot；
- 增加更多 printf 相关字段，哪怕只是格式化；
- 使用软件 sin/cos 代替查表或 CORDIC；
- 增加复杂死区补偿；
- 增加弱磁控制；
- 增加 MTPA/MTPV；
- 增加在线参数辨识；
- 增加更复杂 Observer；
- 增加采样异常统计；
- 增加故障快照字段；
- 在高频中做大量 if/else 诊断。

产品级要求是：每加一个功能，都要记录 DWT 对比，不能只看电机能不能转。

### DWT 与 HSE_VALUE 的关系

DWT 时间换算依赖 CPU 时钟。如果工程中的 SystemCoreClock 或 HSE_VALUE 配置不一致，就可能出现“cycles 是对的，但 us 换算是错的”。因此产品级建议：

- 固定 HSE_VALUE；
- 固定 SystemCoreClock 更新路径；
- DWT 初始化后用已知延时做一次校验；
- 用示波器测 GPIO 翻转周期验证高频真实周期；
- 文档中同时记录 cycles 和 us，不只记录 us。

## 09_硬件保护链路

软件保护是必要的，但不能替代硬件保护。电机功率级的危险故障，例如桥臂直通、相线短路、大电流冲击，必须由硬件链路快速关断。

### 当前工程保护状态

当前 `bsp_tim.c` 中 TIM1 Break 配置为：

```text
BreakState = TIM_BREAK_DISABLE
Break2State = TIM_BREAK2_DISABLE
```

同时代码里启用了 TIM1_BRK_TIM15_IRQn，但由于 BreakState 关闭，当前还不能认为已经形成完整硬件过流保护链路。

当前真正能工作的主要是软件保护，例如：

- 软件电流限值；
- 母线欠压；
- 母线过压；
- 采样 fallback/hold-last 诊断；
- 后续 Observer 失锁、堵转等软件保护。

软件保护最快也要等到下一次 ADC 采样和高频任务执行。按当前 16kHz 计算，最坏情况下软件发现异常可能有接近一个控制周期的延迟：

```text
最坏检测延迟 ≈ 0 ~ 62.5us + ISR 执行时间
```

对于过流硬故障，这个速度不够快。因此产品级必须补硬件 Break。

### 产品级硬件保护推荐链路

推荐的硬件保护链路是：

```text
相电流 / 母线电流 / 驱动器 DESAT / 比较器
  -> 硬件 FAULT 信号
    -> TIM1 BKIN / BKIN2
      -> TIM1 硬件立即关闭 PWM 输出
        -> TIM1 Break 中断
          -> AppFoc 记录故障
            -> 关闭 M1_EN_DRIVER
              -> 进入 FAULT
```

这个链路的关键是：第一步关 PWM 不依赖 CPU 执行软件。CPU 只负责记录和恢复管理。

### Break 后安全态

产品级必须明确 Break 后 TIM1 输出状态：

- 高边是否关闭；
- 低边是否关闭；
- 是否进入 Hi-Z；
- 是否保持 inactive；
- OSSR/OSSI 如何配置；
- MOE 是否被硬件清除；
- AutomaticOutput 是否允许自动恢复；
- 故障清除前是否禁止重新使能驱动。

当前配置：

```text
OffStateRunMode = TIM_OSSR_ENABLE
OffStateIDLEMode = TIM_OSSI_ENABLE
AutomaticOutput = TIM_AUTOMATICOUTPUT_DISABLE
```

这个方向是偏安全的：Break 后不应自动恢复输出。但因为 BreakState 当前关闭，所以还需要后续真正接入 BKIN 并测试。

### 过流阈值与 ADC 量程关系

当前软件过流阈值为 15A，按采样链路计算 ADC 电压变化约 1.8V。如果 offset 在 1.65V，正方向 15A 可能接近 ADC 上限。因此产品级要区分：

- 控制电流限幅；
- 软件过流阈值；
- 硬件比较器过流阈值；
- 驱动器自身过流/短路阈值；
- ADC 饱和阈值。

这些阈值不能全部设成同一个值。推荐关系是：

```text
正常控制最大电流 < 软件限流 < 软件故障阈值 < 硬件快速保护阈值 < 功率器件极限
```

但 ADC 量程必须覆盖软件故障阈值，否则软件还没判断过流，ADC 已经饱和。

### 硬件保护测试要求

产品级至少要测试：

- BKIN 输入拉高/拉低能否立即关断 PWM；
- Break 后 TIM1 CH1/2/3 和 CH1N/2N/3N 的实际电平；
- Break ISR 是否能锁存故障；
- M1_EN_DRIVER 是否关闭；
- 故障清除前是否禁止重新启动；
- 误触发滤波是否合理；
- 故障触发到 PWM 关闭的时间；
- 软件复位后是否保持安全；
- 调试器复位时驱动器是否误使能。

硬件保护测试必须使用限流电源、假负载或安全台架，不要直接在满功率电机上做短路实验。

# 03_控制算法架构

本章用于说明当前工程的控制算法主链路。这里不是单纯解释 FOC 理论，而是按照当前代码的真实模块来写：`foc_current_sense` 负责把 ADC 原始值变成相电流和母线电压；`foc_current_loop` 负责 Clarke/Park、Id/Iq PI、前馈与 dq 限幅；`foc_startup` 负责开环角度和频率爬坡；`foc_observer` 负责封装 `foc_flux_observer_pll`；`foc_handover` 负责开环角到观测角的接管；`foc_svpwm` 负责 SVPWM、CCR 和 ADC 触发点；`app_foc` 只负责把这些模块按状态和周期组织起来。

当前代码已经不是最早期的“固定电压开环”结构，而是逐步靠近商业控制器常见结构：

```text
ADC 原始值
  -> 电流/电压换算
  -> Clarke/Park
  -> Id/Iq 电流环
  -> dq 电压限幅
  -> 可选前馈/解耦
  -> InvPark
  -> 可选死区补偿
  -> SVPWM
  -> TIM1 CCR

同时：
alpha/beta 电压 + alpha/beta 电流
  -> foc_flux_observer_pll
  -> theta_obs / speed_obs

开环启动阶段：
theta_open + theta_obs + open_speed + obs_speed
  -> Handover
  -> theta_ctrl
```

从产品架构看，当前最重要的不是继续堆算法，而是保证每一条链路的输入、输出、单位、坐标系和调用周期都清楚。只要这些边界清楚，后续无论加入弱磁、MTPA、过调制、参数辨识，还是更复杂的 Observer，都不会破坏主链路。

## 01_FOC主控制链路

当前主控制链路运行在 `AppFoc_HighFreqTask()` 中，由 ADC2 注入转换完成中断触发。控制周期由 `FOC_TS_SEC = 6.25e-5s` 定义，对应约 16kHz。每一拍高频控制都应该满足三个要求：输入是本拍采样数据，计算路径固定，输出是本拍 PWM 比较值。

### 当前高频主链路

当前代码中，高频任务根据 `s_state` 分支执行：

```text
OFFSET_CALIB:
  中点 PWM
  关闭驱动
  累计 ADC offset

ALIGN:
  刷新采样
  保护检查
  固定 theta = 0
  Id 电流闭环
  输出静止磁场

OPENLOOP:
  刷新采样
  保护检查
  推进开环角和频率
  Handover 计算接管状态
  选择 theta_ctrl
  运行开环电流模式 / 电压测试 / V/F
  输出 SVPWM
  后台更新 Observer

RUN:
  刷新采样
  保护检查
  使用 theta_obs
  运行 DSoft + FeedForward 电流环
  输出 SVPWM
  后台更新 Observer
```

这里有一个很重要的架构点：当前工程不是让 Observer 直接决定 PWM，也不是让电流环直接切状态。控制角的来源由 `app_foc` 和 `foc_handover` 共同决定，电流环只根据传入的 `theta_ctrl` 计算 `valpha/vbeta`。这种结构比“Observer 内部直接切 RUN”更容易维护。

### 当前控制链路的单位关系

当前工程主要使用以下单位：

- 相电流：A。
- 母线电压：V。
- 机械频率：Hz。
- 机械角速度：rad/s。
- 电角速度：rad/s。
- 电角度：rad。
- 控制周期：s。
- PWM 比较值：CCR 计数。

几个关键换算关系如下：

```text
控制频率：
f_ctrl = 1 / FOC_TS_SEC = 1 / 62.5us = 16000Hz

机械角速度：
omega_m = 2π * f_mech

电角速度：
omega_e = pole_pairs * omega_m

当前 pole_pairs = 4
所以 15Hz 机械频率对应：
omega_m = 2π * 15 = 94.25 rad/s
omega_e = 4 * 94.25 = 377.0 rad/s
```

这也解释了你日志里经常看到 `open_speed_rad_s = 377.0`，它对应的就是 15Hz 机械开环目标频率。

### 当前主链路的优点

当前代码已经具备几个比较好的产品化基础：

- 所有电压输出最终收口到 `app_foc_apply_voltage_command()`。
- `app_foc_apply_voltage_command()` 统一处理死区补偿和 SVPWM。
- `FOC_CurrentLoop` 不直接操作 TIM1。
- `FOC_Observer` 不直接修改状态机。
- `FOC_Handover` 单独负责接管判定。
- `FOC_SpeedLoop` 已经从 `app_foc.c` 抽离出来，速度环只输出 `iq_ref`。
- DWT 统计已经从 `app_foc.c` 抽离到 `app_foc_timing`，避免应用编排文件继续膨胀。

### 当前主链路还需要优化的点

当前代码仍有几个工程级问题，建议在后续版本修正：

- `AppFoc_Stop()` 仍然不应该通过故障入口来做正常停机。正常停机和故障停机要分离，避免正常停机也生成兜底故障。
- `FOC_ObserverIsLocked()` 当前兼容接口返回固定值，接管状态已经迁移到 `FOC_HandoverData_t`，后续应删除该接口，或明确标注只为旧代码兼容。
- `Observer` 在 RUN 中只更新估算，但 RUN 阶段没有完整的失锁保护闭环。后续应将 `handover.locked`、PLL 限幅、速度突变、电流异常共同纳入失锁保护。
- 当前 `foc_config.h` 参数很多，研发阶段可以接受，但产品级应逐步分成板级参数、电机参数、控制参数、保护参数和调试参数。

## 02_电流采样与Clarke_Park

电流采样是整个 FOC 系统的基础。后面所有 Id/Iq、Observer、Handover、保护判断都依赖采样结果。如果采样方向、零偏、比例、扇区重构不对，后面调 PI 和 PLL 基本没有意义。

当前采样模块是 `Control/Sensing/foc_current_sense.c`。它主要做五件事：

```text
1. 读取 ADC2 注入 rank1/rank2/rank3
2. 把 ADC raw 换算成 ADC 引脚电压
3. 减去零偏 offset
4. 按 Rshunt 和放大倍数换算成相电流
5. 根据 SVPWM 扇区选择可信两相并重构第三相
```

### ADC 到电流的换算

当前配置：

```text
FOC_ADC_VREF = 3.3V
FOC_ADC_FULL_SCALE = 4096
FOC_SHUNT_OHM = 0.02Ω
FOC_GAIN = 6.0
```

ADC LSB 电压为：

```text
V_lsb = 3.3 / 4096 = 0.0008057V = 0.8057mV
```

电流采样链路增益为：

```text
Rshunt * Gain = 0.02 * 6 = 0.12V/A
```

所以 1 个 ADC count 对应的电流约为：

```text
I_lsb = 0.0008057 / 0.12 = 0.006714A/count
```

也就是大约 **6.7mA/count**。这对 24V 小功率 PMSM 来说是可以用于电流环调试的分辨率，但实际噪声、运放偏置、PWM 开关噪声会降低有效分辨率。

### 母线电压换算

当前母线电压比例：

```text
FOC_VBUS_RATIO = 25.0
```

所以 ADC raw 到母线电压的换算为：

```text
Vbus = raw * 3.3 / 4096 * 25
```

对应每个 ADC count：

```text
Vbus_lsb = 0.0008057 * 25 = 0.02014V/count
```

也就是约 **20mV/count**。如果母线 24V，则 ADC 引脚电压约为：

```text
Vadc = 24 / 25 = 0.96V
raw ≈ 0.96 / 3.3 * 4096 ≈ 1192
```

因此，如果调试中 24V 母线 raw 大概在 1190 左右，说明母线分压换算基本匹配。

### 零偏校准

当前 `FOC_CURRENT_OFFSET_CALIB_SAMPLES = 4096`，控制频率 16kHz。因此 offset 校准耗时约：

```text
T_offset = 4096 / 16000 = 0.256s
```

也就是约 **256ms**。这和当前启动流程是匹配的：启动后先进入 `OFFSET_CALIB`，高频中断每拍累计一次 ADC 电压，累计完成后由中频状态机切到 `ALIGN`。

零偏校准阶段必须满足：

- PWM 输出中点。
- 驱动器关闭。
- 电机相线没有有效电流。
- ADC 注入采样仍然正常运行。
- 不要在此阶段打开电流环或速度环。

当前 `OFFSET_CALIB` 高频逻辑基本符合这个原则：中点输出、关闭驱动、清空电流环调试量、运行 `FOC_CurrentSense_RunOffsetCalib()`。

### 三低边采样与可信两相

当前硬件按三低边电阻采样设计。三低边采样不是任意时刻都可以可靠读取三相电流，因为某些相在某些 PWM 区间低边不导通，采样电阻上没有对应相电流。当前代码采用“可信两相 + 第三相重构”的结构。

当前映射关系为：

```text
扇区 1/6：V/W 两相可信，U = -(V + W)
扇区 2/3：U/W 两相可信，V = -(U + W)
扇区 4/5：U/V 两相可信，W = -(U + V)
```

这个映射同时存在于：

```text
foc_svpwm_trusted_pair_from_sector()
foc_svpwm_select_trusted_pair_ccr()
foc_current_sense_update_currents()
```

这三个地方必须保持一致。后续如果你修改 SVPWM 扇区定义、相序、硬件相线映射、PWM 极性，这个映射必须重新验证。

### hold-last 采样策略

当前代码支持动态采样失败时的 hold-last：

```text
FOC_ADC_HOLD_LAST_ON_FALLBACK = 1
FOC_ADC_FALLBACK_HOLD_MAX_TICKS = 80
FOC_ADC_FALLBACK_HOLD_FAULT_ENABLE = 0
```

逻辑是：

```text
如果当前 PWM 模块认为采样窗口不足：
  并且上一拍有有效电流：
    OPENLOOP/RUN 中可以沿用上一拍有效电流
```

这个策略适合调试阶段，能避免采样窗口偶发失败导致电流突然跳变。但它也有风险：如果长时间 hold-last，电流环看到的是旧电流，PI 可能继续输出错误电压。

当前代码默认只是记录 `fallback_hold_count`，不直接触发故障。产品级建议分三级处理：

```text
短时间 hold-last：
  允许，记录诊断。

连续 hold-last 超过几十拍：
  降低电流给定或限制电压输出。

连续 hold-last 超过保护阈值：
  触发采样异常故障，进入 FAULT。
```

### Clarke/Park 链路

当前 `FOC_CurrentLoopMeasureDQ()` 中执行：

```text
iu/iv/iw -> Clarke -> ialpha/ibeta
ialpha/ibeta + theta_ctrl -> Park -> id/iq
```

这条链路的关键是 `theta_ctrl`。在不同状态下，`theta_ctrl` 来源不同：

```text
ALIGN:
  theta_ctrl = 0

OPENLOOP:
  theta_ctrl = FOC_HandoverSelectTheta()
  可能是 theta_open
  可能是 theta_open 与 theta_obs 混合
  也可能是 theta_obs

RUN:
  theta_ctrl = FOC_ObserverGetThetaRad()
```

因此，Id/Iq 是否正确，不只取决于采样，还取决于角度来源。如果你看到 `id/iq` 异常，排查顺序应该是：

```text
1. ADC raw 是否合理
2. offset 是否接近中点
3. 相电流符号是否正确
4. 三相和是否接近 0
5. SVPWM 扇区和可信相映射是否正确
6. theta_ctrl 是否方向正确
7. Park 变换角度是否需要补偿
```

## 03_Id_Iq电流环

当前电流环模块为 `Control/Loop/foc_current_loop.c`。它是整个控制算法的核心模块之一，但它不应该承担状态机职责。当前代码总体上已经符合这个方向：电流环函数只接收采样、电角度、Id/Iq 给定和补偿参数，然后输出 `valpha/vbeta`。

### 电流环基本结构

当前常规双电流环路径为：

```text
FOC_CurrentLoopRunWithRef()
  -> FOC_CurrentLoopMeasureDQ()
  -> id_pi
  -> iq_pi
  -> dq 电压矢量限幅
  -> InvPark
  -> valpha/vbeta
```

对应的数学关系可以理解为：

```text
id_err = id_ref - id_meas
iq_err = iq_ref - iq_meas

vd_pi = PI_d(id_err)
vq_pi = PI_q(iq_err)

[v_alpha, v_beta] = InvPark(vd, vq, theta_ctrl)
```

当前代码中还加入了硬件极性修正：

```text
vd_pi = FOC_VD_CMD_SIGN * PI_d(...)
vq_pi = FOC_VQ_CMD_SIGN * PI_q(...)
```

当前配置：

```text
FOC_VD_CMD_SIGN = -1
FOC_VQ_CMD_SIGN = -1
```

这说明你在实测中发现“给正 Vd 得到负 Id、给正 Vq 得到负 Iq”，因此通过符号宏修正输出方向。这个设计比在多个函数里手动取反要好，因为方向问题集中在配置文件里。

### PI 参数与限幅

当前电流环参数：

```text
FOC_IQ_KP = 3.770
FOC_IQ_KI = 3330.0
FOC_ID_KP = FOC_IQ_KP * 0.5
FOC_ID_KI = FOC_IQ_KI * 0.2
FOC_CURR_LOOP_V_LIMIT = 13.5V
FOC_DQ_VOLT_LIMIT = 13.5V
```

电流环有两个层级的限幅：

```text
第一层：
  单轴 PI 输出限幅
  id_pi / iq_pi 的 out_min/out_max 和 integral_min/integral_max

第二层：
  dq 矢量总幅值限幅
  sqrt(vd^2 + vq^2) <= FOC_DQ_VOLT_LIMIT
```

第二层很重要，因为母线电压有限。24V 母线下，SVPWM 线性区最大相电压矢量约为：

```text
Vmax ≈ Vbus / sqrt(3) = 24 / 1.732 = 13.86V
```

当前 `FOC_DQ_VOLT_LIMIT = 13.5V`，基本接近 24V 母线下的 SVPWM 可用电压上限，留了一点余量。这个配置是合理的。

### DSoft 设计

当前 `FOC_CurrentLoopRunDSoft()` 和 `FOC_CurrentLoopRunDSoftFeedForward()` 引入了 d 轴软释放：

```text
d_axis_enable_k: 0 -> 1
vd_cmd_limit_v: 从 1V -> 4V
```

对应配置：

```text
FOC_D_AXIS_SOFT_RAMP_MS = 1000ms
FOC_D_AXIS_VD_LIMIT_START_V = 1.0V
FOC_D_AXIS_VD_LIMIT_END_V = 4.0V
```

它的意义是：开环阶段和刚进入 RUN 时，不让 d 轴 PI 一开始就完全参与。因为无感角度还不一定完全正确，如果 d 轴 PI 很强，可能会把电压推向错误方向，造成抖动或过流。

当前 DSoft 的工程意义可以概括为：

```text
早期：
  更接近 Iq-only
  降低 d 轴误差对电压输出的影响

中期：
  d 轴 PI 逐步参与
  vd 限幅逐步放开

后期：
  恢复完整 Id/Iq 双环
```

这是比较适合当前项目阶段的做法。

### Iq-only 路径

`FOC_CurrentLoopRunIqOnly()` 的作用是只闭 q 轴，d 轴电压保持 0，同时 reset d 轴积分。它适合：

```text
1. 调试 q 轴转矩方向
2. 早期避免 d 轴 PI 干扰
3. 观察速度环只通过 iq_ref 影响转矩
```

不过当前主线已经更多使用 DSoft，而不是纯 Iq-only。产品级可以保留 Iq-only 作为调试模式，不建议作为长期运行模式。

### 开环电压测试模式

`FOC_CurrentLoopRunStartupVoltTest()` 用于固定 dq 电压注入：

```text
FOC_DQ_TEST_VD_V = 1.5V
FOC_DQ_TEST_VQ_V = 0.0V
```

这个模式不是正常运行模式，而是用于验证坐标方向。例如：

```text
给正 Vd：
  看 Id 是正还是负

给正 Vq：
  看 Iq 是正还是负
  看电机转向是否符合预期
```

这个模式对排查方向问题非常有用，建议保留为调试模式。

### 当前电流环还可以优化的点

当前电流环已经能支持无感启动调试，但产品级还可以增强：

- 增加电流环 PI 参数计算说明，明确目标带宽和相位裕度。
- 增加电流环积分冻结条件，例如采样 fallback、母线欠压、电压饱和时的积分处理。
- 增加 dq 电压限幅后的 anti-windup，目前只是 PI 自身输出限幅和积分限幅，矢量限幅后没有把饱和信息反馈给 PI。
- 增加 `voltage_limited` 标志，用于速度环降额和弱磁判断。
- 增加电流环阶跃测试流程，记录 id/iq 上升时间、超调、稳态误差。
- RUN 阶段如果前馈较大，应打印 `vd_pi/vq_pi/vd_ff/vq_ff/vd_cmd/vq_cmd`，判断前馈是在减轻 PI 压力还是在加重 PI 压力。

## 04_速度环

当前速度环已经被抽到 `Control/Loop/foc_speed_loop.c`，这是一个正确方向。速度环本质上属于控制算法层，不应该长期放在 `app_foc.c` 中。`app_foc` 只应该在 RUN 中调用速度环，并把速度环输出的 `iq_ref` 下发给高频控制链。

### 当前速度环输入输出

速度环输入：

```text
observer.speed_rad_s  电角速度
s_medium_task_dt_sec  中频任务时间步长
speed_target_mech_rad_s  目标机械角速度
```

速度环输出：

```text
iq_ref_cmd_a
speed_ref_mech_rad_s
speed_fdb_mech_rad_s
enabled
```

速度环只输出 q 轴电流给定，不直接输出电压，不直接改 PWM，不直接改状态机。

### 电角速度到机械速度

当前电机极对数为 4，因此：

```text
omega_m = omega_e / pole_pairs
```

如果 observer 输出：

```text
omega_e = 377 rad/s
```

则机械速度为：

```text
omega_m = 377 / 4 = 94.25 rad/s
f_mech = 94.25 / 2π = 15Hz
rpm = 15 * 60 = 900rpm
```

这个换算要在文档里写清楚，否则调试速度环时很容易把电角速度和机械速度混用。

### RUN 入口速度环策略

当前 RUN 入口调用 `FOC_SpeedLoopPrepareRunEntry()`，速度环不是立即跳到默认目标，而是先用当前实际速度初始化速度参考。这样做的目的很明确：

```text
OPENLOOP 刚切 RUN：
  observer 速度已经接近开环速度

如果速度环参考突然跳到另一个目标：
  iq_ref 会阶跃
  电流环会突变
  电机可能抖动

所以进入 RUN 时：
  speed_ref 先等于当前速度
  然后按斜坡追目标
```

当前配置：

```text
FOC_SPEED_ENABLE_DELAY_MS = 500ms
FOC_SPEED_REF_RAMP_HZ_PER_S = 3.0Hz/s
FOC_SPEED_IQ_SLEW_A_PER_S = 5.0A/s
FOC_SPEED_TAKEOVER_IQ_HOLD_A = 0.30A
FOC_SPEED_TAKEOVER_HOLD_MS = 300ms
```

这些参数的意义是：RUN 刚进入时先 observer-only hold 一段时间，然后速度环逐步接管，Iq 输出也做斜率限制。

### 当前速度环限幅

当前配置：

```text
FOC_SPEED_IQ_MIN_A = 0.5A
FOC_SPEED_IQ_MAX_A = 7.0A
FOC_SPEED_TARGET_MAX_MECH_RAD_S = 290rad/s
```

`290rad/s` 机械速度换算为 rpm：

```text
rpm = 290 * 60 / 2π ≈ 2770rpm
```

这说明当前速度目标上限还没有完全覆盖 3000rpm。如果产品目标是 3000rpm，机械角速度需要：

```text
omega_m = 3000 / 60 * 2π = 314.16rad/s
```

因此后续如果要目标 3000rpm，`FOC_SPEED_TARGET_MAX_MECH_RAD_S` 至少要大于 314rad/s，同时还要确认 observer 速度限幅、电压余量和前馈是否支撑。

当前 observer 速度限幅：

```text
FOC_OBS_SPEED_LIMIT_RAD_S = 2200rad/s 电角速度
```

对应机械 rpm：

```text
omega_m = 2200 / 4 = 550rad/s
rpm = 550 * 60 / 2π ≈ 5252rpm
```

所以 observer 速度限幅本身能覆盖 3000rpm。真正可能限制 3000rpm 的反而是电压余量、前馈方向、弱磁和速度目标上限。

### 当前速度环建议

当前速度环适合研发调试，但产品级还需要补：

- 速度环正反转支持。
- 制动扭矩支持，目前当前版本倾向只允许正扭矩，避免调试阶段反向拉扯。
- 速度环输出与电流限制联动。
- 电压饱和时速度环 anti-windup。
- 速度反馈异常时冻结速度环。
- 速度目标来自上位机时要做斜坡和合法性检查。
- 速度环调参要有阶跃测试数据，不要只凭耳朵判断。

## 05_开环启动算法

当前开环启动模块为 `Control/Startup/foc_startup.c`，主要负责两个阶段：

```text
ALIGN:
  固定角度，累计 align_ticks

OPENLOOP:
  从起始机械频率爬升到目标机械频率
  推进电角度 theta_e_rad
```

### 当前开环参数

```text
FOC_ALIGN_TIME_MS = 300ms
FOC_ALIGN_ID_REF_A = 0.8A

FOC_OPENLOOP_START_FREQ_HZ = 0.2Hz
FOC_OPENLOOP_TARGET_FREQ_HZ = 15.0Hz
FOC_OPENLOOP_RAMP_TIME_MS = 3000ms
```

对齐时间换算成高频 tick：

```text
align_ticks = 300ms / 62.5us = 4800 ticks
```

开环爬坡时间换算成高频 tick：

```text
ramp_ticks = 3000ms / 62.5us = 48000 ticks
```

目标 15Hz 机械频率对应：

```text
900rpm
377rad/s 电角速度
```

### 当前开环策略

当前 `OPENLOOP` 阶段不是纯电压开环，而是优先使用：

```text
FOC_OPENLOOP_CTRL_MODE = FOC_CTRL_MODE_OPENLOOP_CURRENT
```

也就是：

```text
开环角度 theta_open
  + Id/Iq 电流闭环
```

这个策略比 V/F 更接近 RUN 阶段，因为电流环在 OPENLOOP 中已经工作，RUN 只需要切换角度来源和启用速度环，而不是突然从电压控制切到电流控制。

### 开环阶段的三条路径

当前 `app_foc_run_startup()` 中根据宏选择三条路径：

```text
FOC_CTRL_MODE_OPENLOOP_CURRENT:
  FOC_CurrentLoopRunDSoft()

FOC_CTRL_MODE_OPENLOOP_DQ_VOLT_TEST:
  FOC_CurrentLoopRunStartupVoltTest()

FOC_CTRL_MODE_OPENLOOP_VF:
  FOC_CurrentLoopRunStartupVf()
```

建议定位如下：

```text
OPENLOOP_CURRENT:
  主运行路径，适合逐步产品化。

DQ_VOLT_TEST:
  方向验证路径，适合实验。

VF:
  兜底验证路径，适合确认 PWM/SVPWM 基本输出。
```

### 开环启动的产品级优化

当前开环启动还比较固定，产品级建议补充：

- 启动失败超时。
- 启动失败原因记录。
- 启动重试次数。
- 不同负载下的启动电流曲线。
- 开环频率不是简单线性爬坡，而是 S 曲线或分段曲线。
- 低电压下启动电流自动降额。
- 进入 Handover 前检查电流是否异常、母线是否塌陷。
- 统计启动成功率，例如 100 次启动成功多少次。

## 06_Observer_PLL观测器

当前 Observer 模块为 `Control/Observer/foc_observer.c`，它封装了 `Control/Model/foc_flux_observer_pll.c`。当前重构后，Observer 模块只负责估算，不再负责接管判断，这是正确的边界。

### 当前 Observer 输入

Observer 输入来自本拍控制和采样：

```text
valpha
vbeta
ialpha
ibeta
```

其中：

```text
valpha/vbeta:
  来自电流环输出，并经过可选死区补偿后的实际命令电压

ialpha/ibeta:
  由采样相电流 Clarke 得到
```

Observer 不再传入 `theta_open` 和 `mech_freq_hz`。这说明接管逻辑已经从 Observer 中剥离出去。

### 当前 Observer 输出

Observer 输出：

```text
theta_rad
speed_rad_s
flux angle
flux alpha/beta
PLL error
PLL integral
speed limit flag
integral limit flag
```

这些调试量非常重要。无感问题经常不是“Observer 有没有输出”，而是：

```text
theta 方向是否正确
speed 符号是否正确
PLL 积分是否长期打满
speed 是否被限幅
flux 是否异常跳变
角度是否需要固定补偿
```

### 当前 Observer 参数

```text
FOC_OBS_THETA_COMP_RAD = 2.2003
FOC_OBS_PLL_KP = 120
FOC_OBS_PLL_KI = 3000
FOC_OBS_PLL_INTEGRAL_LIMIT = 1800
FOC_OBS_SPEED_LIMIT_RAD_S = 2200
FOC_OBS_SPEED_FILTER_ALPHA = 0.05
```

其中 `FOC_OBS_THETA_COMP_RAD` 是当前工程非常关键的参数。它代表观测角和控制角之间的相位补偿。这个补偿值不应该凭感觉改，而应该通过 open angle 与 obs angle 的稳定误差来标定。

### Observer 的产品级边界

产品级建议保持这个边界：

```text
Observer 只回答：
  我估计的角度是多少？
  我估计的速度是多少？
  我的 PLL 状态怎么样？
  我的磁链状态怎么样？

Observer 不回答：
  是否切 RUN？
  是否停机？
  是否回退开环？
  是否输出 PWM？
```

这些判断应该交给 Handover、Protection、AppFoc。

### 当前 Observer 需要优化的点

当前 `FOC_ObserverIsLocked()` 和 `FOC_ObserverGetAngleErrDeg()` 已经变成兼容接口，返回固定值或不再承载真实判断。建议后续处理：

```text
方案 A：
  删除旧接口，所有 locked/angle_err 都从 FOC_HandoverGetDebug() 读取。

方案 B：
  保留接口，但明确注释“仅兼容旧代码，不用于接管判断”。

方案 C：
  在 RuntimeSnapshot 中明确拆分：
    observer.theta/speed/pll
    handover.angle_err/ready/blend
```

## 07_Startup到Observer切换

Handover 是当前无感项目最关键的工程环节。它决定从开环角 `theta_open` 过渡到观测角 `theta_obs` 的时机和方式。

当前模块为 `Control/Startup/foc_handover.c`。

### 当前 Handover 输入

```text
FOC_StartupData_t:
  theta_open
  open_speed
  mech_freq_hz

FOC_ObserverData_t:
  theta_obs
  obs_speed

FOC_HandoverData_t:
  hold_ticks
  blend_ticks
  blend_k
  fail_reason
  state
```

### 当前 Handover 判据

当前接管前必须满足：

```text
1. 开环机械频率 >= FOC_SWITCHOVER_MIN_MECH_FREQ_HZ
2. 角度误差 <= FOC_SWITCHOVER_MAX_ANGLE_ERR_DEG
3. 速度误差 <= speed_limit
4. 条件连续满足 FOC_SWITCHOVER_HOLD_MS
```

当前配置：

```text
FOC_SWITCHOVER_MIN_MECH_FREQ_HZ = 10Hz
FOC_SWITCHOVER_MAX_ANGLE_ERR_DEG = 15deg
FOC_SWITCHOVER_MAX_SPEED_ERR_RATIO = 0.60
FOC_SWITCHOVER_MAX_SPEED_ERR_MIN_RAD_S = 20rad/s
FOC_SWITCHOVER_HOLD_MS = 300ms
FOC_SWITCHOVER_BLEND_MS = 800ms
```

换算成高频 tick：

```text
hold_ticks = 300ms / 62.5us = 4800 ticks
blend_ticks = 800ms / 62.5us = 12800 ticks
```

这说明当前切换非常保守，不是某一拍满足就切，而是先连续满足 300ms，再用 800ms 慢慢混合角度。这个对研发阶段是合理的，因为你当前更需要稳定和可解释，而不是最快接管。

### fail_reason 含义

当前 fail_reason 使用位图：

```text
0x01:
  机械频率不足

0x02:
  角度误差超限

0x04:
  速度误差超限
```

调试时不要只看 `sw=0/1`，还要看 `fail_reason`。例如：

```text
fail_reason = 0x02:
  主要是角度对不上
  优先查 theta compensation、方向、Park 角度、Observer 相位

fail_reason = 0x04:
  主要是速度跟不上
  优先查 PLL 参数、速度滤波、开环速度、Observer 输入电压/电流

fail_reason = 0x01:
  开环速度没到最低接管频率
  优先查开环爬坡和目标频率
```

### BLEND 角度混合

当前 `FOC_HandoverSelectTheta()` 中的角度混合不是简单线性插值绝对角，而是先计算最短角差：

```text
delta = WrapPi(theta_obs - theta_open)
theta_ctrl = theta_open + blend_k * delta
```

这个处理非常重要。如果不做最短角差，角度在 0/2π 附近会突然跨越，导致电流环看到大角度阶跃。

### BLEND 中的回退策略

当前 BLEND 状态下，不是一有失败就回退，而是增加了防抖：

```text
如果只是角度轻微毛刺，并且角差 <= FOC_SWITCHOVER_BLEND_MAX_ANGLE_ERR_DEG：
  继续 BLEND

如果速度/频率等严重条件失败：
  连续失败 FOC_SWITCHOVER_BLEND_FAIL_HOLD_TICKS 后才回退
```

当前配置：

```text
FOC_SWITCHOVER_BLEND_MAX_ANGLE_ERR_DEG = 35deg
FOC_SWITCHOVER_BLEND_FAIL_HOLD_TICKS = 20
```

20 拍对应时间：

```text
20 * 62.5us = 1.25ms
```

这相当于允许很短的毛刺，但如果持续失败就回退。

### 当前 Handover 需要优化的点

当前 Handover 已经比早期代码清楚很多，但产品级还需要：

- RUN 后的失锁判定不能只依赖 `handover.state`，还要继续观察角度、速度、电流和 PLL 状态。
- `observer_ready`、`locked`、`ready_now` 三个概念要文档化，避免混用。
- 进入 RUN 后如果失锁，当前会回退 OPENLOOP；产品级要考虑直接 FAULT 还是允许回退，取决于负载和安全需求。
- Handover 参数需要批量启动数据支撑，不能只按一次启动日志调。
- 建议记录每次启动的最大角差、平均速度误差、blend_reset_count、进入 RUN 时间。

## 08_SVPWM调制

当前 SVPWM 模块为 `Control/Modulation/foc_svpwm.c`。它不仅负责把 `valpha/vbeta` 转成三相 CCR，还负责判断当前扇区下的 ADC 采样窗口。

### 当前 SVPWM 计算流程

```text
valpha/vbeta
  -> atan2 得到电压矢量角度
  -> 映射到 sector 1~6
  -> 根据 sector 计算 T1/T2/T0
  -> 七段式中心对齐分配 ta/tb/tc
  -> duty_u/v/w
  -> CCR1/CCR2/CCR3
  -> 计算 TIM1_CH4 ADC 触发点
```

当前 `FOC_Svpwm_ApplySvpwm()` 保存：

```text
svpwm_sector
svpwm_da/db/dc
svpwm_t1/t2/t0
ccr_u/v/w
adc_sample_ccr
adc_window_ok
adc_trigger_fallback
```

这些调试量对采样问题非常关键。

### SVPWM 电压限幅关系

当前电流环限制 `sqrt(vd^2+vq^2) <= 13.5V`。24V 母线下，SVPWM 线性区约 13.86V。因此正常情况下 `T1 + T2` 不应该长期超过 PWM 周期。

如果你看到：

```text
T1 + T2 经常被 scale 回来
或 dq voltage limited 长期为 1
或 vq_cmd 长期贴 13.5V
```

说明当前转速/负载下电压余量不足。此时继续加速度环 PI 不一定有用，需要考虑：

```text
1. 反电动势前馈方向
2. 电流环 PI 是否被电压饱和
3. 母线电压是否不足
4. 是否需要弱磁
5. 电机参数是否错误
```

### 动态 ADC 采样点

当前 PWM 模块根据可信两相 CCR 计算采样窗口：

```text
safe_start = max(ccr_x, ccr_y) + guard
safe_end = FOC_ADC_SAMPLE_MAX_CCR
safe_width = safe_end - safe_start
```

如果中点在安全窗口中，继续使用中点；否则移动到窗口中间。如果窗口不足，则 fallback 到 `FOC_ADC_TRIG_FALLBACK_CCR`，并设置 `adc_trigger_fallback = 1`。

当前配置：

```text
FOC_ADC_TRIG_MARGIN_CCR = 140
FOC_ADC_TRIG_MIN_WINDOW_CCR = 320
FOC_ADC_TRIG_FALLBACK_CCR = 2655
FOC_ADC_DYNAMIC_SAMPLE_ENABLE = 1
```

按 62.5us / 5311 估算，一个 CCR tick 约 11.77ns，则：

```text
guard 140 tick ≈ 1.65us
min window 320 tick ≈ 3.77us
```

这个窗口包括开关瞬态避让、死区、采样放大器建立和 ADC 采样转换余量。产品级应通过示波器和电流波形确认这些数字，而不是只靠软件估算。

### 当前 SVPWM 可以优化的点

- 当前扇区判断使用 `atan2f`，可读性好但耗时较高。若 DWT 余量不足，可改为基于比较器的扇区判断。
- 当前 T1/T2 使用 `sinf`，可读性好但也有耗时。后续可替换为更高效的几何公式或查表。
- 当前过调制只是简单压缩 T1/T2，产品级应明确是否允许过调制，以及过调制区如何影响采样窗口和电流环。
- 当前 ADC 触发点与三低边采样已关联，但还需要实测验证窗口是否真的干净。

## 09_死区补偿

当前死区补偿模块为 `Control/foc_deadtime_comp.c`。当前配置中总开关是打开的：

```text
FOC_DT_COMP_ENABLE = 1
```

但三个阶段默认关闭：

```text
FOC_DT_COMP_ENABLE_ALIGN = 0
FOC_DT_COMP_ENABLE_OPENLOOP = 0
FOC_DT_COMP_ENABLE_RUN = 0
```

这说明当前工程把死区补偿作为预留能力，但为了隔离实验，暂时不让它影响主控制链路。这个决策是合理的。因为无感项目中，如果采样、Observer、Handover 还没有完全稳定，过早打开死区补偿会增加变量数量。

### 死区补偿的本质

死区会导致相电压误差，误差方向与相电流方向相关。简化理解：

```text
电流为正：
  实际相电压可能比指令低或高，取决于桥臂定义

电流为负：
  误差方向相反
```

因此死区补偿一般根据三相电流方向产生补偿电压，再转成 alpha/beta 叠加到控制电压上。

### 当前配置

```text
FOC_DT_COMP_K_BASE = 0.016
FOC_DT_COMP_K_MAX = 0.020
FOC_DT_COMP_CURR_TH_A = 0.30A
FOC_DT_COMP_MIN_VBUS_V = 8.0V
```

其中 `FOC_DT_COMP_CURR_TH_A` 用于过零平滑，避免电流接近 0 时补偿方向频繁跳变。

### 建议打开顺序

不要一次性在所有阶段打开。建议顺序：

```text
1. 保持 ALIGN/OPELOOP/RUN 都关闭，先稳定启动和切换。
2. 只在 RUN 中打开，观察 id/iq 波动、噪声和 vq_pi。
3. 如果 RUN 有改善，再考虑 OPENLOOP。
4. ALIGN 阶段通常不建议打开，除非确认对齐过程需要补偿。
```

### 如何判断死区补偿有效

不要只听声音。建议观察：

```text
id/iq 纹波是否减小
vq_pi 平均值是否更合理
同一速度下电流噪声是否下降
电机低速是否更平顺
相电流过零附近是否有尖峰
```

如果打开后噪声变大，优先检查 `FOC_DT_COMP_SIGN`。

## 10_前馈与解耦补偿

当前前馈补偿已经在 `FOC_CurrentLoopRunDSoftFeedForward()` 中实现，位置是 PI 输出之后、dq 限幅之前。这是合理位置。

### 当前前馈结构

当前支持两类补偿：

```text
1. 反电动势前馈：
   vq_bemf ≈ omega_e * flux

2. 交叉耦合补偿：
   vd_dec ≈ -omega_e * Lq * iq
   vq_dec ≈  omega_e * Ld * id
```

当前代码会先按逻辑 dq 方程计算，再乘以 `FOC_VD_CMD_SIGN / FOC_VQ_CMD_SIGN` 和前馈方向修正。

### 当前前馈参数

```text
FOC_FF_ENABLE = 1
FOC_BEMF_FF_ENABLE = 1
FOC_DECOUPLE_FF_ENABLE = 1

FOC_BEMF_FF_GAIN_TARGET = 0.50
FOC_DECOUPLE_FF_GAIN_TARGET = 0.10

FOC_FF_MIN_ELEC_SPEED_RAD_S = 100
FOC_FF_MIN_VBUS_V = 10
FOC_BEMF_FF_RAMP_MS = 300
FOC_DECOUPLE_FF_DELAY_MS = 300
FOC_DECOUPLE_FF_RAMP_MS = 500
```

这说明当前设计比较保守：低速不补，母线太低不补，补偿从 0 缓慢爬升，交叉耦合还要延迟。

### 反电动势补偿计算示例

当前磁链：

```text
FOC_FLUX_WB = 0.0078Wb
```

如果电角速度为 377rad/s，则理论反电动势项约：

```text
Vbemf = omega_e * flux = 377 * 0.0078 = 2.94V
```

如果补偿系数为 0.5，则补偿约：

```text
Vbemf_ff = 1.47V
```

这不是小量，尤其在 24V 母线、dq 限幅 13.5V 的系统里，它会明显影响 q 轴电压分配。

如果电机到 3000rpm：

```text
机械频率 = 50Hz
电角速度 = 2π * 50 * 4 = 1256.6rad/s
Vbemf = 1256.6 * 0.0078 = 9.80V
50% 前馈 = 4.90V
```

因此高速下反电动势前馈很重要，但方向错了会非常明显地恶化控制。

### 如何验证前馈效果

不要只看电机能不能转。建议打印：

```text
vq_pi
vq_ff_raw
vq_ff
vq_cmd
vd_pi
vd_ff
vd_cmd
voltage_limit
iq_ref
iq_meas
speed
```

如果前馈方向正确，通常会看到：

```text
同一速度下：
  vq_pi 负担下降
  iq 跟踪误差下降
  电压限幅减少
  速度跟随更快
```

如果方向错误，通常会看到：

```text
vq_pi 反而更大
iq 误差增大
电机噪声变大
速度上不去
电压更快饱和
```

### 当前前馈建议

- 先单独打开反电动势前馈，交叉耦合系数保持小。
- 先从 0.1、0.2、0.3、0.5 逐步加，不要一步给 1.0。
- 每次只改一个参数，并固定测试速度。
- 用日志判断 `vq_pi` 是否下降，而不是只凭体感。
- 前馈不是弱磁，不能无限降低电压需求；超过基速仍然需要弱磁或提高母线电压。

## 11_弱磁控制预留

当前工程尚未真正实现弱磁控制，但已经具备加入弱磁的基础：

```text
1. 有 id_ref 入口
2. 有 dq 电压矢量限幅
3. 有母线电压测量
4. 有速度反馈
5. 有前馈补偿
6. 有速度目标管理
```

### 弱磁触发条件

弱磁不应该按速度固定触发，而应该优先看电压余量：

```text
如果 sqrt(vd^2 + vq^2) 接近 Vlimit
并且速度仍低于目标
并且 iq 已经接近电流限制
则逐步给负 id
```

也就是说，弱磁是电压受限后的补救，不是低速也一直给负 id。

### 初版弱磁建议

初版可以做最简单的电压余量型弱磁：

```text
Vmag = sqrt(vd^2 + vq^2)

如果 Vmag > 0.95 * Vlimit:
  id_ref -= small_step

如果 Vmag < 0.85 * Vlimit:
  id_ref += small_step 向 0 恢复

id_ref 限制在 [id_fw_min, 0]
总电流 sqrt(id^2 + iq^2) 不超过电流限制
```

### 当前项目是否急需弱磁

如果目标是先稳定到 2000~2500rpm，并解决 Handover 和速度跟随，弱磁不是第一优先级。优先级更高的是：

```text
1. 采样干净
2. 电流环稳定
3. Observer 角度方向正确
4. Handover 稳定
5. 反电动势前馈方向正确
6. 速度环不饱和
```

等这些稳定后，再加入弱磁。

```text
  vq_pi 是否下降、限幅是否减少、速度是否改善

死区：
  电流谐波是否降低、过零尖峰是否减小
```

产品级算法不是“代码里有功能”就算完成，而是要有可复现的测试数据证明它有效。


# 04_状态机架构

本章说明当前工程的状态机架构。这里重点写清楚三件事：

```text
1. 状态机有哪些状态
2. 高频任务在每个状态干什么
3. 中频任务在每个状态干什么
```

当前工程的状态机不是单纯的 `switch-case`，而是分成了两个时间尺度：

```text
高频任务：
  由 ADC2 注入完成中断触发，约 16kHz。
  负责本拍实时控制，包括采样、保护、电流环、Observer、PWM 输出。

中频任务：
  在 main while 中运行，约 1ms。
  负责慢速状态推进、速度环、命令缓冲、运行快照。
```

这种结构很重要。FOC 控制里最怕把所有事情都放到高频中断，导致中断路径越来越长、越来越不稳定。当前架构把“快执行”和“慢决策”拆开，是向产品级工程靠拢的正确方向。

## 01_主状态机总览

当前主状态定义在 `FOC_StateTypeDef`，主要包括：

```text
FOC_STATE_IDLE
FOC_STATE_OFFSET_CALIB
FOC_STATE_ALIGN
FOC_STATE_OPENLOOP
FOC_STATE_RUN
FOC_STATE_FAULT
```

当前主状态由 `app_foc.c` 中的 `s_state` 保存。原则上，只有 App 层应该修改主状态。算法模块只输出事实、判据或控制结果，不应该直接把系统切到 RUN 或 FAULT。

### 当前状态流

当前正常启动流程：

```text
IDLE
  -> AppFoc_Start()
OFFSET_CALIB
  -> offset_done
ALIGN
  -> align_ticks >= align_ticks_target
OPENLOOP
  -> handover.state == OBSERVER
RUN
```

当前异常流程：

```text
ALIGN / OPENLOOP / RUN
  -> FOC_ProtectionCheckFault() != FOC_FAULT_NONE
FAULT
```

当前停机流程：

```text
任意运行状态
  -> AppFoc_Stop()
IDLE
```

注意：当前代码中 `AppFoc_Stop()` 仍然通过故障入口做停机，这一点建议后续修正。正常停机不应该生成故障快照，也不应该写入兜底故障码。

### 状态机分层原则

当前状态机应该遵守：

```text
高频：
  不做复杂业务判断
  不解析上位机命令
  不打印
  不写 Flash
  不做大结构体长期处理
  只做本拍必须完成的实时动作

中频：
  状态推进
  速度环
  快照刷新
  命令提交
  按键/上位机目标更新

调试：
  只读取快照
  不直接修改状态机
```

## 02_IDLE状态

IDLE 是安全空闲状态。

### IDLE 的目标

```text
PWM 输出中点
驱动器关闭
电流环调试量清零
不运行电流环
不运行 Observer 接管
不运行速度环
等待启动命令
```

### 高频任务在 IDLE 干什么

当前 `AppFoc_HighFreqTask()` 中：

```text
case FOC_STATE_IDLE:
  app_foc_force_mid_output()
  app_foc_clear_current_loop_debug()
```

也就是说，高频侧只保证输出安全，不做控制。

### 中频任务在 IDLE 干什么

当前 `app_foc_medium_handle_state_machine()` 中：

```text
theta_mode = OPENLOOP
speed_loop_enable = 0
id_ref = 0
iq_ref = 0
speed_ref = 0

如果刚进入 IDLE：
  FOC_HandoverReset()
  FOC_CurrentLoopReset()
  FOC_StartupResetAlign()
```

中频侧的重点是一次性清理控制对象，避免下一次启动带着旧状态。

### IDLE 的产品级优化

产品级 IDLE 不应该只是“什么都不干”，还可以做：

```text
参数合法性检查
母线电压就绪检查
驱动器故障引脚检查
温度检查
启动权限检查
上位机连接状态检查
```

但这些不应该放在高频中断里，而应该放在中频或低频任务中。

## 03_OFFSET_CALIB状态

OFFSET_CALIB 是电流零偏校准状态。

### OFFSET_CALIB 的目标

```text
驱动关闭
PWM 中点
电机无有效电流
累计 ADC offset
得到 iu_offset_v / iv_offset_v / iw_offset_v
```

当前配置：

```text
FOC_CURRENT_OFFSET_CALIB_SAMPLES = 4096
FOC_TS_SEC = 62.5us
```

所以校准时间约：

```text
4096 * 62.5us = 256ms
```

### 高频任务在 OFFSET_CALIB 干什么

当前高频逻辑：

```text
app_foc_force_mid_output()
app_foc_clear_current_loop_debug()
app_foc_drive_enable(0)
FOC_CurrentSense_RunOffsetCalib(&s_current_sense, &next_state)

如果 next_state == ALIGN:
  s_feedback.offset_done = 1
```

注意这里高频任务没有直接把 `s_state` 改为 ALIGN，而是只设置 `s_feedback.offset_done`。真正状态切换交给中频任务。这是比较好的设计。

### 中频任务在 OFFSET_CALIB 干什么

中频逻辑：

```text
theta_mode = ALIGN
speed_loop_enable = 0

如果刚进入 OFFSET_CALIB:
  FOC_HandoverReset()
  FOC_CurrentLoopReset()
  FOC_StartupResetAlign()

如果 s_feedback.offset_done != 0:
  FOC_StartupResetAlign()
  s_state = ALIGN
```

也就是说：

```text
高频负责采样事实：
  offset 是否完成

中频负责状态推进：
  完成后切 ALIGN
```

### OFFSET_CALIB 的风险点

需要注意：

```text
1. 校准期间驱动必须关闭。
2. 校准期间不要让电流环输出有效电压。
3. offset raw 应该接近运放中点。
4. 三相 offset 差异不应过大。
5. 如果母线采样未准备好，vbus 默认 24V，不应影响 offset。
```

### 产品级优化

产品级应加入 offset 合法性检查：

```text
offset 是否在 0.3V~3.0V 范围
三相 offset 差值是否超过阈值
校准期间 raw 抖动是否过大
校准是否超时
校准失败禁止启动
```

## 04_ALIGN状态

ALIGN 是转子对齐状态。

### ALIGN 的目标

```text
打开驱动器
固定电角度
给 d 轴小电流
让转子吸到已知角度附近
为后续开环启动提供初始角度
```

当前参数：

```text
FOC_ALIGN_TIME_MS = 300ms
FOC_ALIGN_ID_REF_A = 0.8A
FOC_ALIGN_THETA_OFFSET_RAD = 0
```

高频 tick 计算：

```text
align_ticks_target = 300ms / 62.5us = 4800 ticks
```

### 高频任务在 ALIGN 干什么

当前 `app_foc_run_align()` 执行：

```text
如果 align_ticks == 0:
  app_foc_drive_enable(1)

app_foc_current_sense_update()
FOC_ProtectionCheckFault()

theta_e = 0
FOC_CurrentLoopRunWithRef(
  theta = 0,
  id_ref = FOC_ALIGN_ID_REF_A,
  iq_ref = 0
)

强制 vq_cmd = 0
InvPark 得到 valpha/vbeta
app_foc_apply_voltage_command()
align_ticks++
```

这说明当前对齐不是固定电压对齐，而是固定角度下的 d 轴电流闭环对齐。这比固定电压更可控，因为对齐电流受 PI 管理。

### 为什么对齐阶段强制 q 轴电压为 0

当前代码在 `FOC_CurrentLoopRunWithRef()` 后又把 `vq_cmd` 清为 0。这是为了避免对齐阶段产生额外转矩。

对齐阶段理想目标是：

```text
只建立静止磁场
不让电机持续旋转
```

如果 q 轴电压或 q 轴电流参与，电机可能出现持续抖动或轻微转动，影响后续开环起步。

### 中频任务在 ALIGN 干什么

中频逻辑：

```text
theta_mode = ALIGN
speed_loop_enable = 0

如果刚进入 ALIGN:
  FOC_StartupResetAlign()

如果 align_ticks >= align_ticks_target:
  FOC_StartupResetRun()
  FOC_HandoverReset()
  FOC_CurrentLoopReset()
  app_foc_d_axis_soft_reset()
  s_run_entry_prepared = 0
  s_state = OPENLOOP
```

也就是说：

```text
高频负责持续对齐并累计 align_ticks。
中频负责判断时间到没到。
```

### ALIGN 的调试重点

如果 ALIGN 阶段有问题，优先看：

```text
id_ref 是否为 0.8A
id_meas 是否能跟住
iq_meas 是否接近 0
vd/vq 是否过大
是否触发过流
对齐时电机是否剧烈抖动
```

如果对齐时震动大，可能原因：

```text
电流方向反
theta 对齐角不合适
id PI 太强
offset 不准
采样噪声大
电机负载卡滞
```

## 05_OPENLOOP状态

OPENLOOP 是无感启动最关键的状态。

### OPENLOOP 的目标

```text
让电机从静止附近逐步加速到 Observer 可工作的速度
让 Observer 在后台跟踪
让 Handover 判断是否可以接管
尽量降低从开环角切到观测角的冲击
```

当前开环目标：

```text
起始机械频率：0.2Hz
目标机械频率：15Hz
爬坡时间：3000ms
```

15Hz 机械频率对应：

```text
900rpm
377rad/s 电角速度
```

### 高频任务在 OPENLOOP 干什么

当前 `app_foc_run_startup()` 执行：

```text
cmd = app_foc_get_active_command()

app_foc_current_sense_update()
FOC_ProtectionCheckFault()

FOC_StartupAdvance(&s_startup)

FOC_HandoverUpdate(&s_handover, &s_startup, &s_observer)

theta_ctrl = FOC_HandoverSelectTheta(
  &s_handover,
  &s_startup,
  &s_observer
)

根据 FOC_OPENLOOP_CTRL_MODE 选择控制方式：
  OPENLOOP_CURRENT:
    app_foc_d_axis_soft_update()
    FOC_CurrentLoopRunDSoft()

  DQ_VOLT_TEST:
    FOC_CurrentLoopRunStartupVoltTest()

  VF:
    FOC_CurrentLoopRunStartupVf()

app_foc_apply_voltage_command()

FOC_ObserverUpdate(&s_observer, &s_current_sense, valpha, vbeta)
```

这条链路非常重要。它说明 OPENLOOP 阶段同时做了三件事：

```text
1. 继续用开环角控制电机
2. 后台运行 Observer
3. Handover 持续评估能不能接管
```

### 中频任务在 OPENLOOP 干什么

中频逻辑：

```text
theta_mode = OPENLOOP
speed_loop_enable = 0

如果刚进入 OPENLOOP:
  FOC_HandoverReset()
  FOC_CurrentLoopReset()
  FOC_StartupResetRun()
  app_foc_d_axis_soft_reset()
  s_run_entry_prepared = 0

如果 s_handover.state == FOC_SW_STATE_OBSERVER:
  app_foc_prepare_run_entry()
  s_state = RUN
```

这里中频只关心一个结果：Handover 是否已经完成接管。真正接管过程在高频里通过 `FOC_HandoverUpdate()` 和 `FOC_HandoverSelectTheta()` 每拍执行。

### OPENLOOP 中的 theta_ctrl

OPENLOOP 的控制角不是固定只用开环角。当前可能有三种情况：

```text
Handover 状态为 OPENLOOP:
  theta_ctrl = theta_open

Handover 状态为 BLEND:
  theta_ctrl = theta_open + blend_k * shortest_delta(theta_obs - theta_open)

Handover 状态为 OBSERVER:
  theta_ctrl = theta_obs
```

也就是说，OPENLOOP 状态内部已经包含了切换过程。主状态只有在 Handover 完成后才切 RUN。

### OPENLOOP 的调试重点

OPENLOOP 不成功通常分为三类：

```text
电机本身没跟上开环：
  iq 不够
  startup ramp 太快
  负载太大
  align 不稳

Observer 没跟上：
  valpha/vbeta 输入不对
  ialpha/ibeta 方向不对
  PLL 参数不合适
  theta compensation 不对

Handover 不给接管：
  角差超限
  速度误差超限
  hold 时间不够
  blend 中回退
```

对应要看：

```text
open_speed_rad_s
obs_speed_rad_s
speed_err_rad_s
angle_err_deg
fail_reason
blend_k
blend_reset_count
id/iq
vd/vq
```

## 06_SWITCHOVER过程

当前 SWITCHOVER 不是独立主状态，而是 OPENLOOP 内部的子状态机，由 `FOC_HandoverData_t.state` 表示。

### Handover 子状态

当前包括：

```text
FOC_SW_STATE_OPENLOOP
FOC_SW_STATE_BLEND
FOC_SW_STATE_OBSERVER
```

### OPENLOOP 子状态

目标：

```text
只使用开环角
连续判断 Observer 是否满足接管条件
```

判据：

```text
机械频率 >= 10Hz
角差 <= 15deg
速度误差在允许窗口内
条件持续 300ms
```

如果条件断掉：

```text
hold_ticks 清零
observer_ready = 0
blend_k = 0
```

### BLEND 子状态

目标：

```text
从 theta_open 平滑过渡到 theta_obs
```

混合时间：

```text
FOC_SWITCHOVER_BLEND_MS = 800ms
blend_ticks = 12800 ticks
```

控制角：

```text
delta = WrapPi(theta_obs - theta_open)
theta_ctrl = theta_open + blend_k * delta
```

如果 BLEND 中出现轻微角度毛刺，并且角差不超过 `FOC_SWITCHOVER_BLEND_MAX_ANGLE_ERR_DEG = 35deg`，代码允许继续过渡，不立即回退。

如果严重条件持续失败：

```text
blend_fail_ticks >= 20
  -> reset handover
  -> 回到 OPENLOOP 子状态
```

### OBSERVER 子状态

目标：

```text
Observer 完全接管角度
blend_k = 1
observer_ready = 1
```

当 `s_handover.state == FOC_SW_STATE_OBSERVER` 时，中频状态机把主状态切到 RUN。

### Handover 与 RUN 的关系

主状态切 RUN 的触发条件是：

```text
s_handover.state == FOC_SW_STATE_OBSERVER
```

切 RUN 之前，中频会先调用：

```text
app_foc_prepare_run_entry()
```

它会准备速度环入口，避免 RUN 首拍速度环冲击。

### Handover 的产品级建议

当前 Handover 已经比较完整，但还可以改进：

```text
1. 把每次启动的接管数据保存到启动诊断结构。
2. 区分 locked、ready_now、observer_ready、state 的含义。
3. RUN 后继续做失锁监测。
4. 如果 RUN 后失锁，决定是回 OPENLOOP 还是 FAULT。
5. Handover 参数通过批量启动测试确定。
```

## 07_RUN状态

RUN 是 Observer 接管后的闭环运行状态。

### RUN 的目标

```text
使用 theta_obs 作为控制角
运行完整 Id/Iq 电流环
启用速度环输出 iq_ref
启用前馈补偿
保持可观测、可保护、可调速
```

### 高频任务在 RUN 干什么

当前 `app_foc_run_run()` 执行：

```text
cmd = app_foc_get_active_command()

app_foc_current_sense_update()
FOC_ProtectionCheckFault()

theta_ctrl = FOC_ObserverGetThetaRad(&s_observer)

app_foc_d_axis_soft_update()

FOC_CurrentLoopRunDSoftFeedForward(
  sampling,
  theta_ctrl,
  cmd.id_ref_a,
  cmd.iq_ref_a,
  d_axis_enable_k,
  vd_limit,
  observer_speed,
  vbus
)

app_foc_apply_voltage_command()

FOC_ObserverUpdate(&s_observer, &s_current_sense, valpha, vbeta)
```

RUN 高频链路的重点是：

```text
角度来源：Observer
Iq 给定来源：中频速度环
前馈：RUN 中启用
电压输出：仍然统一经过 app_foc_apply_voltage_command()
```

### 中频任务在 RUN 干什么

中频逻辑：

```text
theta_mode = OBSERVER
id_ref = 0

如果刚进入 RUN 且入口未准备：
  app_foc_prepare_run_entry()

FOC_SpeedLoopStep(
  observer.speed_rad_s,
  s_medium_task_dt_sec
)

读取 speed_snapshot:
  speed_loop_enable
  speed_ref_mech_rad_s
  iq_ref_cmd_a

s_cmd.iq_ref_a = speed_snapshot.iq_ref_cmd_a

如果 s_handover.state != OBSERVER:
  回退 OPENLOOP
```

这说明速度环运行在中频，不在 16kHz 高频 ISR 中运行。速度环输出 `iq_ref`，通过 `s_cmd_active` 被高频读取。

### RUN 入口准备

`app_foc_prepare_run_entry()` 做了关键动作：

```text
FOC_SpeedLoopPrepareRunEntry(
  observer speed,
  startup electrical speed
)

s_run_entry_prepared = 1

把速度目标同步到按键模块
```

它的作用是让速度环从当前速度平滑接管，不要一进 RUN 就把目标速度阶跃到默认值。

### RUN 中回退 OPENLOOP

当前代码中：

```text
if (s_handover.state != FOC_SW_STATE_OBSERVER)
{
  reset startup / handover / d_axis_soft
  s_state = OPENLOOP
}
```

这是当前研发阶段的最小回退策略。但产品级要谨慎。因为 RUN 后电机已经进入闭环，如果 observer 失锁，直接回 OPENLOOP 不一定安全。产品级建议根据应用决定：

```text
小功率实验台：
  可尝试回 OPENLOOP

高可靠产品：
  更建议进入 FAULT，记录失锁快照
```

### RUN 的调试重点

RUN 阶段重点看：

```text
speed_ref
speed_fdb
iq_ref
iq_meas
id_meas
vd_pi/vq_pi
vd_ff/vq_ff
vd_cmd/vq_cmd
theta_obs
PLL error
voltage limited
fault_flags
DWT max cycles
```

如果速度跟不上，排查顺序：

```text
1. speed target 是否被限制
2. iq_ref 是否达到上限
3. vq 是否达到电压上限
4. observer speed 是否正确
5. 前馈方向是否正确
6. 母线电压是否足够
7. 是否需要弱磁
```

## 08_FAULT状态

FAULT 是故障锁定状态。

### FAULT 的目标

```text
停止有效 PWM 输出
关闭驱动
锁存故障快照
防止自动重新启动
等待人工或上位机清故障
```

### 故障进入

当前故障进入由 `app_foc_enter_fault()` 收口。它应该负责：

```text
如果 s_fault_flags == FOC_FAULT_NONE:
  写入兜底故障

app_foc_latch_fault_snapshot()
FOC_Svpwm_SetTim1Mid()
app_foc_drive_enable(0)
s_state = FAULT
```

这比让 `foc_protection.c` 直接关 GPIO 更好，因为故障动作属于 App 层统一收口。

### 高频任务在 FAULT 干什么

当前高频中：

```text
case FOC_STATE_FAULT:
  app_foc_clear_current_loop_debug()
  app_foc_enter_fault()
```

这个能保证即使反复进入高频，也维持故障停机状态。但需要注意 `app_foc_enter_fault()` 内部应避免重复覆盖第一次故障快照。当前 `app_foc_latch_fault_snapshot()` 有 `valid` 判断，只锁存第一次故障，这是正确的。

### 中频任务在 FAULT 干什么

中频中：

```text
theta_mode = OPENLOOP
speed_loop_enable = 0
id_ref = 0
iq_ref = 0
speed_ref = 0

如果刚进入 FAULT:
  FOC_CurrentLoopReset()
  FOC_HandoverReset()
```

也就是说，中频负责把命令清到安全值，并清理控制状态。

### 当前 FAULT 需要优化的点

- 正常 `AppFoc_Stop()` 不应调用 `app_foc_enter_fault()`。
- FAULT 恢复条件还不完整，需要明确是按键清故障、上位机清故障，还是重新上电。
- 应区分软件保护故障、硬件 Break、采样异常、Observer 失锁、ISR 超时。
- 故障快照后续应支持最近 N 次持久化。

## 09_启动流程

当前启动流程由 `AppFoc_Start()` 触发。

### AppFoc_Start 做了什么

当前启动动作：

```text
启动 TIM1 CH1/2/3 PWM
启动 TIM1 CH1/2/3 互补 PWMN
启动 TIM1 Base
启动 ADC1 regular
设置 ADC1 ready
启动 ADC1 injected
启动 ADC2 injected interrupt

reset observer
reset deadtime
reset handover
reset current loop
reset startup align/run
清 offset 累计
reset sampling runtime
清 feedback
清 fault flags
清 fault snapshot
reset timing stats
reset speed loop
reset d-axis soft
PWM 中点
关闭驱动
s_state = OFFSET_CALIB
更新 runtime snapshot
```

这个启动流程比较完整，适合研发阶段。

### 启动后的状态推进

```text
AppFoc_Start()
  -> s_state = OFFSET_CALIB

高频累计 offset
  -> s_feedback.offset_done = 1

中频发现 offset_done
  -> s_state = ALIGN

高频 ALIGN 累计 align_ticks
  -> s_feedback.align_ticks 更新

中频发现 align_ticks 到目标
  -> reset startup/current/handover
  -> s_state = OPENLOOP

高频 OPENLOOP 持续运行 handover
  -> handover.state = OBSERVER

中频发现 handover.state == OBSERVER
  -> app_foc_prepare_run_entry()
  -> s_state = RUN
```

### 启动流程的产品级优化

产品级启动前应增加 PRECHECK：

```text
参数 CRC 正常
母线电压在范围内
驱动器无故障
温度正常
ADC offset 未异常
通信命令合法
上一次故障允许恢复
```

当前工程可以先不实现完整 PRECHECK，但文档和架构上要预留。

## 10_停机流程

当前停机由 `AppFoc_Stop()` 实现。

### 当前停机动作

当前代码大致流程：

```text
app_foc_enter_fault()

停止 ADC injected
停止 ADC regular
停止 TIM1 base
停止 PWMN
停止 PWM
设置 ADC1 not ready
s_state = IDLE
s_fault_flags = NONE
reset deadtime
reset command
commit command
update runtime snapshot
```

### 当前停机的问题

这里有一个明确的架构问题：

```text
正常停机调用了 app_foc_enter_fault()
```

这会导致：

```text
正常停机也可能生成故障快照
正常停机与故障停机混淆
调试时无法区分用户 Stop 和真实 Fault
```

建议改成两个内部函数：

```text
app_foc_shutdown_outputs():
  FOC_Svpwm_SetTim1Mid()
  app_foc_drive_enable(0)

app_foc_enter_fault():
  latch fault snapshot
  app_foc_shutdown_outputs()
  s_state = FAULT
```

然后：

```text
AppFoc_Stop():
  app_foc_shutdown_outputs()
  stop ADC/TIM/PWM
  s_state = IDLE
```

### 产品级停机分类

产品级停机建议分三类：

```text
正常停机：
  速度/电流斜坡降到 0
  进入 IDLE

快速停机：
  关闭速度环
  快速拉低 iq_ref
  到低速后关 PWM

故障停机：
  立即关闭输出
  锁存故障
  进入 FAULT
```

当前工程可以先保留快速停机，但正常 Stop 和 Fault 必须分清。

## 11_故障恢复流程

当前故障恢复比较简单，主要通过重新启动流程恢复。研发阶段这样是安全的，因为自动恢复可能导致电机反复冲击。

### 当前恢复前必须清理的对象

恢复前应至少清理：

```text
FOC_CurrentLoopReset()
FOC_ObserverReset()
FOC_HandoverReset()
FOC_StartupResetAlign()
FOC_StartupResetRun()
FOC_SpeedLoopReset()
FOC_CurrentSense_ResetRuntime()
fault_flags
fault_snapshot
d_axis_soft
```

当前 `AppFoc_Start()` 已经做了大部分清理。

### 产品级恢复策略

产品级应按故障类型设计：

```text
欠压：
  电压恢复后允许自动恢复或等待启动命令

过压：
  等待电压下降，通常需要人工确认

软件过流：
  停机锁定，需要人工确认

硬件 Break：
  必须人工确认，可能需要重新上电或驱动器复位

采样异常：
  重新 offset 或禁止启动

Observer 失锁：
  根据应用决定回退、重启或故障锁定

ISR 超时：
  记录并停机，避免继续运行
```

### 故障恢复的架构原则

不要从 FAULT 直接跳 RUN。恢复必须从安全链路重新开始：

```text
FAULT
  -> 清故障
IDLE
  -> Start
OFFSET_CALIB
  -> ALIGN
  -> OPENLOOP
  -> RUN
```

## 12_状态机与命令接口

当前状态机和高频控制之间通过 `s_cmd` 和 `s_cmd_active` 解耦。

### 命令结构

当前 `AppFoc_Command_t` 包含：

```text
state
theta_mode
id_ref_a
iq_ref_a
speed_ref_mech_rad_s
speed_loop_enable
```

中频修改 `s_cmd`，然后通过 `app_foc_commit_command_active()` 原子提交给高频侧。高频通过 `app_foc_get_active_command()` 读取。

### 为什么需要 s_cmd_active

因为高频在中断中运行，中频在 main while 中运行。如果中频正在改命令，高频同时读取，可能读到半更新数据。当前代码通过关中断提交完整结构体，避免了这个问题。

```text
中频：
  准备 s_cmd
  关中断
  s_cmd_active = s_cmd
  恢复中断

高频：
  读取 s_cmd_active 的一份局部拷贝
```

这属于一个简单但有效的双缓冲命令机制。

### 当前命令接口的不足

当前命令结构还偏内部。产品级可以扩展为外部命令接口：

```text
Motor_Start()
Motor_Stop()
Motor_ClearFault()
Motor_SetSpeed()
Motor_SetTorque()
Motor_SetMode()
Motor_SaveParam()
Motor_LoadParam()
```

外部命令先进入命令队列或命令缓冲，由中频状态机验证合法性，再提交给高频，而不是外部直接改 `s_state`。

# 05_保护与故障架构

## 01_故障分类总览

故障应按来源分为硬件故障、采样故障、控制故障、算法失锁、实时性故障、通信/命令故障和参数故障。当前工程已有 fault_flags 基础，但产品级需要完整故障码体系。

故障分类的目的不是“多列几个名字”，而是决定故障动作：立即硬件关断、软件快速停机、降额运行、禁止启动、允许自动恢复、必须人工复位。

- **设计原则**：危险故障优先硬件关断，软件负责诊断、记录和恢复管理。
- **验收方式**：通过故障注入测试验证每个故障码是否能触发正确动作并留下可解释快照。

## 02_软件保护

当前软件保护主要在高频任务中调用 FOC_ProtectionCheckFault，对采样结果和运行状态做检查。软件保护适合检测持续性异常，例如母线欠压/过压、电流超过软件限值、采样 hold-last 超限。

软件保护不能替代硬件过流，因为软件保护依赖 ADC 采样、ISR 执行和代码路径。真正危险的过流必须由硬件 Break 先关断。

- **设计原则**：危险故障优先硬件关断，软件负责诊断、记录和恢复管理。
- **验收方式**：通过故障注入测试验证每个故障码是否能触发正确动作并留下可解释快照。

## 03_硬件保护

硬件保护应独立于 CPU 主循环和控制算法，即使软件死机也能关断功率输出。典型链路是采样电阻/驱动器/比较器 -> TIM1 BKIN -> PWM 强制关闭 -> 中断记录故障。

产品级硬件保护必须验证触发延时、阈值误差、误触发滤波、Break 后输出状态和恢复条件。

- **设计原则**：危险故障优先硬件关断，软件负责诊断、记录和恢复管理。
- **验收方式**：通过故障注入测试验证每个故障码是否能触发正确动作并留下可解释快照。

## 04_过流保护

过流保护分为硬件快速过流和软件电流限值。当前配置 FOC_CURRENT_LIMIT_A 提供软件限流/保护阈值，实际产品还应有更快的硬件过流。

调试时应区分“电流环正常响应中的短时峰值”和“失控过流”。软件保护可加入持续时间滤波，但硬件保护应快速动作。

- **设计原则**：危险故障优先硬件关断，软件负责诊断、记录和恢复管理。
- **验收方式**：通过故障注入测试验证每个故障码是否能触发正确动作并留下可解释快照。

## 05_母线欠压过压保护

母线电压直接影响 SVPWM 可用电压、电流环稳定性和 Observer 估算。当前配置有 FOC_VBUS_UNDERVOLTAGE_V 和 FOC_VBUS_OVERVOLTAGE_V。

欠压时电压指令可能饱和，Observer 误差增加，电机容易失步；过压时功率级和母线电容风险增加。产品级应加入启动前电压检查、运行中欠压降额和过压快速停机。

- **设计原则**：危险故障优先硬件关断，软件负责诊断、记录和恢复管理。
- **验收方式**：通过故障注入测试验证每个故障码是否能触发正确动作并留下可解释快照。

## 06_采样异常保护

采样异常包括 ADC raw 饱和、offset 漂移、三相电流和异常、连续 fallback、连续 hold-last、电流跳变不合理、ADC overrun。当前工程已经有 fallback_request、used_hold_last_current 和 fallback_hold_count，为采样异常保护提供基础。

产品级不要让采样异常静默进入电流环，否则 PI 会基于错误反馈输出更大电压，导致问题扩大。

- **设计原则**：危险故障优先硬件关断，软件负责诊断、记录和恢复管理。
- **验收方式**：通过故障注入测试验证每个故障码是否能触发正确动作并留下可解释快照。

## 07_Observer失锁保护

Observer 失锁是无感控制特有风险。表现为 theta_obs 与真实角度偏离、speed_obs 异常、PLL 积分打满、speed_limit_hit、angle_err 增大、切换后电流突变。

当前工程已有 locked、angle_err、pll_err、pll_integral、speed_limit_hit 等观测量，后续应形成失锁判据：角差持续过大、速度突变、PLL 限幅持续、iq/id 异常等。

- **设计原则**：危险故障优先硬件关断，软件负责诊断、记录和恢复管理。
- **验收方式**：通过故障注入测试验证每个故障码是否能触发正确动作并留下可解释快照。

## 08_堵转与失步保护

堵转与失步在无感启动阶段尤其常见。堵转可能表现为速度上不去、iq 持续较大、电角度推进但机械不转、Observer 速度异常；失步可能表现为电流突变、速度估计跳变、角差增大。

产品级应在 OPENLOOP 和 RUN 分别设计堵转判据。OPENLOOP 可看开环时间、电流和 Observer 速度；RUN 可看速度误差、iq_ref、iq_meas、角差和功率。

- **设计原则**：危险故障优先硬件关断，软件负责诊断、记录和恢复管理。
- **验收方式**：通过故障注入测试验证每个故障码是否能触发正确动作并留下可解释快照。

## 09_ISR超时保护

ISR 超时会破坏控制周期，导致采样、PWM 更新和电流环延迟不稳定。当前 DWT 统计提供了 overrun_count 基础。

产品级应把 ISR 超时纳入故障体系。一次偶发超时可记录，连续超时应降级或停机。调试输出、复杂浮点、Flash 写入和协议处理不得进入高频中断。

- **设计原则**：危险故障优先硬件关断，软件负责诊断、记录和恢复管理。
- **验收方式**：通过故障注入测试验证每个故障码是否能触发正确动作并留下可解释快照。

## 10_故障等级与动作

故障等级建议分为提示、降额、软件停机、硬件锁死四级。不同等级对应不同动作和恢复策略。

例如通信超时可能降额或停机；母线欠压可等待恢复；软件过流应停机；硬件 Break 应锁死并要求人工确认。

- **设计原则**：危险故障优先硬件关断，软件负责诊断、记录和恢复管理。
- **验收方式**：通过故障注入测试验证每个故障码是否能触发正确动作并留下可解释快照。

## 11_故障快照

当前 FOC_FaultSnapshot_t 已经记录 state、fault_flags、id/iq、vd/vq、valpha/vbeta、theta、speed、vbus、sector、fallback 等关键数据。这个方向非常重要。

产品级应将故障快照持久化到 Flash 或外部存储，并记录时间戳、复位原因、最近运行状态、温度、母线电压、最大电流和故障前若干帧数据。

- **设计原则**：危险故障优先硬件关断，软件负责诊断、记录和恢复管理。
- **验收方式**：通过故障注入测试验证每个故障码是否能触发正确动作并留下可解释快照。

## 12_故障恢复策略

故障恢复不能简单地从 FAULT 跳回 RUN。必须先关闭输出、清 PI、清 Observer、清 Handover、重新校准或确认采样，然后从安全状态重新启动。

产品级恢复策略应按故障类型区分：欠压可自动恢复，过温需冷却，过流需人工确认，参数非法禁止启动。

- **设计原则**：危险故障优先硬件关断，软件负责诊断、记录和恢复管理。
- **验收方式**：通过故障注入测试验证每个故障码是否能触发正确动作并留下可解释快照。

## 13_产品级保护待补项

待补保护包括硬件 Break 完整链路、温度保护、参数非法保护、通信超时保护、上电自检失败保护、Observer 失锁保护、堵转保护、采样异常保护、ISR 超时保护和故障持久化。

保护系统不能只在异常发生后停机，还应帮助定位原因。每个保护都要有触发条件、滤波时间、动作、恢复条件和对应快照字段。

- **设计原则**：危险故障优先硬件关断，软件负责诊断、记录和恢复管理。
- **验收方式**：通过故障注入测试验证每个故障码是否能触发正确动作并留下可解释快照。

# 06_参数与标定架构

## 01_参数分类总览

当前参数主要集中在 foc_config.h 中，以宏的形式存在。研发阶段这样方便编译调试，但产品级需要逐步区分编译期参数、板级标定参数、电机参数、控制参数、保护参数和调试参数。

参数分类的意义是确定谁能改、何时能改、是否需要保存、是否需要校验。比如 PWM ARR 属于硬件结构参数，不应运行时随意改；速度目标可以运行时改；电流采样比例应通过标定写入 Flash。

- **参数原则**：先保证硬件参数正确，再调控制参数；先离线保存，再上线运行。
- **产品化要求**：每个参数都应有单位、默认值、范围、来源、修改权限和生效时机。

## 02_电机参数

电机参数包括极对数、Rs、Ld、Lq、磁链、额定电流、额定转速、反电动势常数、最大允许温度等。当前配置中已有 FOC_POLE_PAIRS、FOC_RS_OHM、FOC_LS_H、FOC_FLUX_WB。

这些参数会直接影响 Observer、前馈、解耦、速度换算和弱磁。产品级应提供参数识别或标定流程，并记录参数来源和测试条件。

- **参数原则**：先保证硬件参数正确，再调控制参数；先离线保存，再上线运行。
- **产品化要求**：每个参数都应有单位、默认值、范围、来源、修改权限和生效时机。

## 03_板级参数

板级参数包括 Rshunt、采样放大倍数、ADC Vref、母线分压比例、PWM ARR、死区时间、ADC 通道映射、电流方向符号、驱动器极性。当前 foc_config.h 已包含 FOC_SHUNT_OHM、FOC_GAIN、FOC_VBUS_RATIO、FOC_CURR_SIGN_U/V/W 等。

# 07_调试与观测架构

## 01_调试系统总览

当前工程已经具备比较好的快照调试思路：高频控制对象不直接被外部读取，而是由 AppFoc 汇总 RuntimeSnapshot、TraceSnapshot、FaultSnapshot 和 HfTimingSnapshot。

调试系统的目标不是“打印越多越好”，而是在不破坏实时性的前提下，快速判断问题属于采样、角度、电流环、PWM、Observer、切换、速度环还是保护。

- **实时性原则**：高频中断只写轻量快照，不打印、不格式化、不做协议。
- **调试原则**：每个调试量都应说明来源、单位、刷新周期和用途。

## 02_运行快照RuntimeSnapshot

RuntimeSnapshot 面向整体运行观察，包含 state、fault_flags、svpwm_sector、startup、observer、sampling、current_loop、speed_loop、handover 等结构。

当前 AppFoc_GetRuntimeSnapshot 使用 seq 双读方式降低中频/低频读写撕裂风险，这是很好的工程习惯。产品级可继续扩展但要避免无限膨胀。

- **实时性原则**：高频中断只写轻量快照，不打印、不格式化、不做协议。
- **调试原则**：每个调试量都应说明来源、单位、刷新周期和用途。

## 03_高频调试快照TraceSnapshot

TraceSnapshot 更偏底层，适合排查 PWM 扇区、采样窗口、CCR、ADC raw、三相电流和重构质量。当前默认可通过 FOC_HF_DEBUG_SNAPSHOT_ENABLE 控制。

高频快照不能每拍打印，只能每拍写内存，低频读取或通过上位机抽样输出。

- **实时性原则**：高频中断只写轻量快照，不打印、不格式化、不做协议。
- **调试原则**：每个调试量都应说明来源、单位、刷新周期和用途。

## 04_故障快照FaultSnapshot

FaultSnapshot 用于记录故障发生瞬间的关键上下文。当前已经包含 id/iq、vd/vq、theta、speed、vbus、sector、fallback 等，非常适合故障复盘。

产品级应将 FaultSnapshot 持久化，并支持上位机读取最近故障记录。

- **实时性原则**：高频中断只写轻量快照，不打印、不格式化、不做协议。
- **调试原则**：每个调试量都应说明来源、单位、刷新周期和用途。

## 05_DWT实时性观测

DWT 观测用于统计高频 ISR 执行时间。当前记录 cycles 而不是在高频中换算 us，避免增加高频开销。

调试时应同时看 last、max、avg、overrun_count。只有 max 明显小于周期预算，才说明还有提高 PWM 频率或增加算法的余量。

- **实时性原则**：高频中断只写轻量快照，不打印、不格式化、不做协议。

## 10_硬件出厂测试

硬件出厂测试包括电源输入、驱动使能、三相 PWM、ADC raw、母线电压、相电流 offset、温度通道、通信接口、按键、DAC、FAULT 输入和硬件 Break。

出厂测试应自动生成报告，记录硬件版本、固件版本、参数版本和测试结果。

- **测试输出**：每个测试都应输出数据、结论、参数版本、固件版本和硬件版本。
- **回归要求**：修改控制算法、采样、PWM、Observer、保护或参数系统后，必须重新跑相关回归测试。

## 11_参数烧录与标定

参数烧录与标定应纳入生产流程。烧录内容包括固件、默认参数、电机参数、板级标定参数和产品序列号。

标定完成后应写入 CRC，启动时校验参数，不合法则禁止启动或恢复默认。

- **测试输出**：每个测试都应输出数据、结论、参数版本、固件版本和硬件版本。
- **回归要求**：修改控制算法、采样、PWM、Observer、保护或参数系统后，必须重新跑相关回归测试。

## 12_Bootloader与固件升级预留

当前工程还未完整包含 Bootloader，但产品级需要考虑固件升级。升级系统应支持固件校验、版本管理、断电保护、回滚和升级失败恢复。

电机控制固件升级尤其要防止升级过程中误启动功率级。Bootloader 阶段应保持驱动关闭。

- **测试输出**：每个测试都应输出数据、结论、参数版本、固件版本和硬件版本。
- **回归要求**：修改控制算法、采样、PWM、Observer、保护或参数系统后，必须重新跑相关回归测试。

## 13_产品级验收标准

产品级验收不应只写“电机能转”。应量化为启动成功率、速度误差、电流环响应、温升、保护动作时间、故障记录完整性、实时性余量、长时间运行时间和生产一致性。

验收标准必须和测试用例绑定，否则很难判断工程是否真的达到可交付。

- **测试输出**：每个测试都应输出数据、结论、参数版本、固件版本和硬件版本。
- **回归要求**：修改控制算法、采样、PWM、Observer、保护或参数系统后，必须重新跑相关回归测试。

# 附录B：推荐的后续重构优先级

- **第一优先级**：稳定主控制链：采样可信、电流方向正确、电流环稳定、Observer 角速度可信、Handover 成功率提升。
- **第二优先级**：完善观测：RuntimeSnapshot 字段收敛、FaultSnapshot 持久化、DWT 统计、DAC 输出、上位机基础帧。
- **第三优先级**：完善保护：硬件 Break、采样异常、Observer 失锁、堵转、ISR 超时、温度保护。
- **第四优先级**：参数系统：参数合法性检查、Flash 存储、CRC、默认参数、上位机读写。
- **第五优先级**：产品测试：启动成功率、长时间运行、负载扰动、极限电压、出厂测试、固件升级。
